# 用户管理模块后端开发提示词

## 模块目标

在现有身份与权限体系之上，补齐一个最小但完整的用户管理域，支持管理员维护用户、用户组和成员关系，并保证停用、删除、最后一个有效 ADMIN 保护等关键规则在服务端生效。

## 本期边界

1. 只做用户管理与用户组管理。
2. 不做权限中心，不做角色权限编辑器。
3. 不做菜单权限配置，不让用户组承载后端权限。
4. 不做自助注册、邀请注册、邮箱激活、密码找回、SSO、MFA。
5. 角色集合在创建用户时可指定，后续编辑页只读展示，本期不做管理员改角色流程。
6. 审计只做落库，不单独做审计页面。

## 服务端数据模型

至少包含以下对象：

1. `UserGroup`
   - `id`
   - `name`，唯一
   - `description`
   - `created_at`
   - `updated_at`
2. `UserGroupMember`
   - `group_id`
   - `user_id`
   - `created_at`
   - 同一 `group_id + user_id` 唯一
3. `UserAdminAuditLog`
   - `actor_user_id`
   - `action`
   - `entity_type`
   - `entity_id`
   - `before_json`
   - `after_json`
   - `meta_json`
   - `created_at`

继续复用现有：

1. `User`
2. `UserRole`
3. `AuthSession`
4. `RefreshToken`
5. `UserStatus`
6. `RoleCode`

## 固定接口

### 用户接口

1. `GET /api/v1/users`
2. `POST /api/v1/users`
3. `GET /api/v1/users/{id}`
4. `PATCH /api/v1/users/{id}`
5. `POST /api/v1/users/{id}/status`
6. `DELETE /api/v1/users/{id}`

### 用户组接口

1. `GET /api/v1/user-groups`
2. `POST /api/v1/user-groups`
3. `GET /api/v1/user-groups/{id}`
4. `PATCH /api/v1/user-groups/{id}`
5. `DELETE /api/v1/user-groups/{id}`
6. `POST /api/v1/user-groups/{id}/members`
7. `DELETE /api/v1/user-groups/{id}/members/{user_id}`

## 接口口径

### 创建用户

1. 必填：`username`、`password`、`role_codes`
2. `role_codes` 至少一个
3. 可选：`display_name`、`email`、`group_ids`
4. 创建成功默认 `status=active`

### 编辑用户

1. 只允许修改：
   - `display_name`
   - `email`
   - `group_ids`
2. `username` 创建后不可修改
3. 本期不提供角色编辑

### 用户状态变更

1. `POST /users/{id}/status` 只接受：
   - `active`
   - `disabled`
2. 不在 UI 暴露 `pending`
3. 停用时必须：
   - 校验目标当前为 `active`
   - 校验目标不是最后一个有效 ADMIN
   - 更新状态为 `disabled`
   - 写入 `disabled_at` / `disabled_reason`
   - `token_version += 1`
   - 撤销全部活动会话与 refresh token
   - 记录审计日志
4. 启用时必须：
   - 仅允许从 `disabled -> active`
   - 清理 `disabled_at` / `disabled_reason`
   - 不恢复历史会话，用户需重新登录

### 删除用户

1. 只允许删除从未参与任何业务的用户
2. 删除前必须校验目标不是最后一个有效 ADMIN
3. 删除成功时同步清理用户组关系
4. 删除不存在用户时必须返回明确失败，不静默成功

### 用户组

1. 用户组只维护 `name`、`description`
2. 组名唯一
3. 只允许删除空组
4. 如果组内仍有成员，删除必须失败并返回明确原因

### 成员关系

1. 添加成员前需校验组存在、用户存在
2. 一个用户可以属于多个组
3. 重复入组必须返回冲突，不产生重复关系
4. 移除不存在的成员关系必须返回明确失败

## 权限规则

1. 所有接口都必须在后端校验当前激活角色是否为 `ADMIN`
2. 非 `ADMIN` 必须返回 `403`
3. 不能只依赖前端隐藏入口

## 最后一个有效 ADMIN 保护

统一规则：

1. 有效 ADMIN = `User.status = active` 且拥有启用中的 `UserRole.role_code = ADMIN`
2. 停用前要校验
3. 删除前也要校验
4. 系统中必须始终保留至少一个有效 ADMIN

## 业务参与检查

实现一个集中服务，例如 `has_business_participation(user_id)`，只要命中任意留痕即判定“已参与业务”，阻止删除。

至少检查以下方向：

1. 工单主表与工单活动
2. 知识库内容与点赞
3. 报告与模板产物
4. 任务模块实例与执行日志
5. 实时通知 / 协作留痕
6. 认证安全事件 / 会话 / refresh token

只有用户组关系不应视为业务参与。

## 实现要求

1. 关键写操作必须写 `UserAdminAuditLog`
2. 列表接口支持前端筛选：
   - 用户：关键词、状态、角色、用户组
   - 用户组：关键词
3. 列表和详情输出必须满足前端表格与抽屉使用
4. 所有关键约束应尽量在单次事务内完成，避免半成功状态

## 前端适配要求

前端 `/users` 页面需要：

1. 用户列表表格
2. 用户组列表表格
3. 用户详情 / 编辑抽屉
4. 用户组详情 / 编辑抽屉
5. 删除、停用、启用确认提示
6. 失败原因可直接展示给管理员

因此后端错误响应不要只返回模糊文本，应确保能稳定区分：

1. 权限错误
2. 资源不存在
3. 唯一性冲突
4. 状态非法
5. 最后一个有效 ADMIN 保护
6. 已参与业务不可删除
7. 用户组非空不可删除
8. 重复入组 / 关系不存在

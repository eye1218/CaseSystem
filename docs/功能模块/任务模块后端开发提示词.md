# 任务模块后端开发提示词

你现在要为一个工单系统设计并实现 **任务模块（Tasks Module）后端**。
请严格按照以下要求理解和输出，不要自行扩展业务边界，不要把任务模块做成通用流程引擎，也不要把 Event、模板、任务执行历史三者混成一张抽象大表。

---

## 1. 模块目标

任务模块的目标是：

1. 提供 **任务模板维护能力**；
2. 提供 **任务实例查询能力**；
3. 支持管理员维护 Email / Webhook 两类任务模板；
4. 在 Event 命中后，为每个绑定模板独立创建任务实例；
5. 异步执行任务，并记录执行日志与结果摘要；
6. 支持对失败任务执行人工重试。

本期任务模块是：

- 任务模板管理模块
- 任务实例运行态管理模块
- 任务执行日志查询模块

本期任务模块不是：

- 自动化编排平台
- 通用工作流引擎
- 多步骤 DAG 执行器
- 带自动补偿与依赖管理的任务系统

---

## 2. 功能边界

### 2.1 本期需要实现

1. 任务模板创建；
2. 任务模板编辑；
3. 任务模板启用 / 停用；
4. 任务模板列表查询；
5. 任务模板详情查询；
6. 任务实例列表查询；
7. 任务实例详情查询；
8. 任务失败后人工重试；
9. Event 命中时独立生成任务实例；
10. 异步执行 Email / Webhook 任务；
11. 写入任务执行日志；
12. 保存模板快照，保证历史可追溯。

### 2.2 本期禁止自行扩展

1. 不实现自动重试；
2. 不实现取消任务；
3. 不实现任务依赖关系；
4. 不实现多步骤编排；
5. 不实现任务优先级调度器；
6. 不实现 Email / Webhook 之外的第三种任务类型；
7. 不实现模板层对 Webhook 请求内容的覆盖；
8. 不实现客户可见的任务执行日志；
9. 不实现任务实例软删除；
10. 不实现批量重试；
11. 不实现通用“目标对象选择器”。

---

## 3. 关键概念区分

### 3.1 任务模板

任务模板是可被 Event 绑定的配置对象，绑定的是：

```text
task_template_id
```

它不是任务实例。

### 3.2 任务实例

任务实例是 Event 实际命中后生成的运行态记录。

任务实例必须保存：
- 当前模板引用
- 模板快照
- 执行状态
- 结果摘要
- 错误信息

注意：
- 模板后续被停用或删除，历史实例仍然必须可查看
- 因此实例必须保存模板快照，不能只依赖实时 join 当前模板

### 3.3 Event 与任务模块边界

Event 模块只负责：

1. 判断规则是否命中；
2. 计算应触发的时机；
3. 为每个绑定模板调用任务模块创建任务实例。

Event 模块不负责：

1. 执行任务；
2. 记录任务日志；
3. 展示任务执行详情。

### 3.4 Webhook 模板边界

Webhook 任务模板本期只维护：

- 名称
- 状态
- 引用的 Webhook 渲染模板
- 固定目标对象配置

禁止在任务模板层维护：
- 自定义 URL
- 自定义 Method
- 自定义 Headers
- 自定义 Body

这些内容必须完全来自引用模板。

---

## 4. 固定类型

### 4.1 TaskType

固定为：

```text
EMAIL
WEBHOOK
```

### 4.2 TaskStatus

固定为：

```text
PENDING
RUNNING
SUCCESS
FAILED
CANCELLED
```

注意：
- 本期虽然不实现主动取消，但状态枚举中保留 `CANCELLED`
- 不要额外设计 `RETRYING`、`SKIPPED`、`BLOCKED`

### 4.3 RecipientSourceType

固定为：

```text
CUSTOM_EMAIL
CURRENT_HANDLER
ROLE_MEMBERS
```

---

## 5. 数据模型

本期固定包含以下三类核心模型。

### 5.1 TaskTemplate

字段建议：

1. id
2. name
3. description
4. task_type
5. status
6. template_id
7. target_config_json
8. to_recipients_json
9. cc_recipients_json
10. bcc_recipients_json
11. created_by
12. updated_by
13. created_at
14. updated_at

说明：
- `template_id` 指向现有渲染模板模块中的 email / webhook 模板
- `target_config_json` 对 Webhook 来说，本期固定表达“来源 Event 的当前工单上下文”
- Email 才真正使用 `to/cc/bcc`

### 5.2 TaskInstance

字段建议：

1. id
2. task_template_id，可为空
3. task_template_snapshot_json
4. source_event_rule_id，可为空
5. source_event_binding_id，可为空
6. source_event_code
7. ticket_id，可为空
8. status
9. latest_result_summary
10. error_message
11. started_at
12. finished_at
13. triggered_by
14. retry_of_task_id，可为空
15. created_at
16. updated_at

说明：
- `task_template_id` 可为空，用于支持模板已删除或绑定失效后的历史查看
- `task_template_snapshot_json` 必须保留创建实例时的模板关键配置

### 5.3 TaskExecutionLog

字段建议：

1. id
2. task_instance_id
3. phase
4. input_snapshot_summary
5. render_summary
6. provider_response_summary
7. error_message
8. actor
9. created_at

说明：
- 这是执行链路日志，不是审计总线
- 重点承载每个阶段的关键摘要，便于前端时间线展示

### 5.4 RecipientRule 是否拆表

本期可选：

- 可拆成独立表
- 也可统一存入 JSON

如果没有明显收益，优先使用 JSON，避免无意义的表膨胀。

---

## 6. 对外接口

固定需要提供：

### 6.1 任务模板接口

1. `GET /api/v1/task-templates`
2. `POST /api/v1/task-templates`
3. `GET /api/v1/task-templates/{id}`
4. `PATCH /api/v1/task-templates/{id}`
5. `POST /api/v1/task-templates/{id}/status`

### 6.2 任务实例接口

1. `GET /api/v1/tasks`
2. `GET /api/v1/tasks/{id}`
3. `POST /api/v1/tasks/{id}/retry`

### 6.3 Event 集成接口

Event 模块现有：

1. `GET /api/v1/events/task-templates`

这里必须改为返回真实且可绑定的任务模板，而不是硬编码占位数据。

---

## 7. 权限规则

### 7.1 任务模板

- 仅 `ADMIN` 可创建、编辑、启停任务模板
- 非管理员不可访问模板维护接口

### 7.2 任务工作台

- 内部角色可查看任务实例列表与详情
- 客户不可查看任务实例工作台
- 客户不可查看执行日志
- 客户不可执行重试

### 7.3 对象级访问

如果系统已存在对象级访问控制，应复用现有语义，不额外设计新的 ACL 子系统。

---

## 8. 执行链路

任务执行状态流转固定为：

```text
PENDING -> RUNNING -> SUCCESS / FAILED
```

### 8.1 Email 任务

执行步骤：

1. 读取任务模板；
2. 解析 `to/cc/bcc` 收件规则；
3. 基于工单上下文解析 `CURRENT_HANDLER` 或 `ROLE_MEMBERS`；
4. 渲染 Email 模板；
5. 调用 SMTP 发送；
6. 写执行日志；
7. 更新任务实例状态。

### 8.2 Webhook 任务

执行步骤：

1. 读取任务模板；
2. 验证引用模板类型与目标配置；
3. 渲染 Webhook 模板；
4. 使用 HTTP 客户端调用；
5. 写执行日志；
6. 更新任务实例状态。

补充规则：
- 如果 HTTP method 是 `GET`，发送时忽略 body
- 但渲染结果中仍允许保留 body，便于日志查看
- HTTP method 枚举直接复用现有模板模块的 `GET/POST/PUT/PATCH/DELETE`

### 8.3 失败处理

以下情况都应落为 `FAILED` 并写日志：

1. 模板缺失；
2. 模板已停用；
3. 模板渲染失败；
4. 收件规则非法；
5. Webhook URL 非法；
6. SMTP 发送失败；
7. Webhook 请求失败；
8. Event 绑定失效。

注意：
- 一个 Event 命中多个绑定时，不允许因为其中一个失败而阻塞其他绑定

---

## 9. Event 集成要求

### 9.1 查询可绑定模板

`GET /api/v1/events/task-templates` 必须查询真实任务模板，返回：

- 真实 task_template_id
- 名称
- 类型
- 状态

### 9.2 命中规则后的行为

`dispatch_event_binding` 不再返回占位数据。

正确行为是：

1. Event 命中某个 binding；
2. 任务模块为该 binding 独立创建一条 `TaskInstance`；
3. 异步执行；
4. 成功或失败都写入日志和实例状态。

### 9.3 历史绑定降级

若 Event 规则已绑定的任务模板后来停用、删除或无效：

1. Event 详情仍要能展示历史绑定信息；
2. 运行时仍创建失败实例；
3. 错误信息要明确体现失效原因；
4. 不要直接静默跳过。

---

## 10. 重试规则

重试能力固定为人工重试。

规则如下：

1. 仅 `FAILED` 任务允许重试；
2. 重试必须新建一个新的任务实例；
3. 新实例通过 `retry_of_task_id` 指向原实例；
4. 原失败记录不可被覆盖；
5. 若已存在同源的 `PENDING` 或 `RUNNING` 重试实例，则拒绝再次重试；
6. 不实现自动退避；
7. 不实现最大重试次数策略。

---

## 11. 运行时依赖

本期最小运行时能力需要补齐：

1. SMTP 配置；
2. Webhook timeout 配置；
3. `httpx` 作为正式运行时依赖。

配置建议至少包含：

1. SMTP host
2. SMTP port
3. SMTP username
4. SMTP password
5. SMTP from address
6. SMTP use TLS / SSL
7. webhook timeout seconds

---

## 12. 校验要求

### 12.1 Email 任务模板校验

必须校验：

1. 引用模板存在且类型正确；
2. 收件规则类型合法；
3. `CUSTOM_EMAIL` 必须提供合法邮箱；
4. `ROLE_MEMBERS` 必须提供角色；
5. 至少有一个 `to` 规则。

### 12.2 Webhook 任务模板校验

必须校验：

1. 引用模板存在且类型正确；
2. 不允许在任务模板层提交 URL / method / headers / body 覆盖字段；
3. 目标对象配置只能使用当前 Event 工单上下文。

### 12.3 重试校验

必须校验：

1. 原任务状态必须为 `FAILED`；
2. 当前操作者必须有内部角色权限；
3. 不允许并发重复重试。

---

## 13. 测试要求

测试必须覆盖：

1. 管理员可维护 Email / Webhook 任务模板；
2. 非管理员访问模板接口被拒绝；
3. Email 收件规则合法性校验；
4. Webhook 模板不允许覆盖请求内容；
5. Event 命中多个绑定时独立生成任务实例；
6. Email 成功执行；
7. Webhook 成功执行；
8. 模板渲染失败；
9. 非法 URL 失败；
10. `GET + body` 发送时忽略 body；
11. 任务列表按权限与筛选条件返回；
12. 失败任务重试创建新实例；
13. 重复点击重试被拒绝。

---

## 14. 实现约束总结

请牢记以下约束，不要自行扩展：

1. 本期只有 Email / Webhook 两类任务；
2. 本期没有自动重试；
3. 本期没有取消任务；
4. Webhook 请求内容只能来自引用模板；
5. 任务实例必须保存模板快照；
6. Event 与任务执行状态必须解耦；
7. 重试必须新建实例，而不是覆盖原实例；
8. 客户不可见任务运行态与执行日志；
9. 优先复用现有权限、模板、HTTP method 枚举与基础设施。

# Figma 多页面系统提示词

本文档整理 SOC 7x24 工单系统的 Figma 多页面高保真设计提示词，用于生成整套前端后台与协作门户界面。现有 [Figma工单详情页提示词.md](/Volumes/data/workspace/python/CaseSystem/DOCS/Figma工单详情页提示词.md) 可继续作为详情页专项深化版本使用。

## 提示词

```text
请为一个 “SOC 7x24 工单系统” 设计一套企业级 Web 管理后台与协作门户界面，面向安全运营中心（SOC）场景。系统服务于 7x24 值守、安全事件响应、分层协同、SLA 管控、工单池流转、升级处理、审计留痕、客户反馈与闭环协作。

本次设计要求采用“多页面系统版”输出，不是单一页面设计。请输出一整套页面结构清晰、导航统一、组件复用一致的企业级后台界面。整套设计必须属于同一套设计系统，但每个模块页面要有明确边界和独立的页面职责。

当前阶段范围约束：
- 不设计 Event 与动作编排页面
- 不设计模板页面
- 不设计外部集成详细页面
- 不设计可观测性增强详细页面
- 可观测性当前仅保留管理员菜单或二级入口占位，不展开具体界面

一、总体设计目标

整体产品定位应偏向：
- 安全运营中台 / 企业级响应平台
- 高信息密度
- 强状态感、时效压迫感、责任归属感
- 快速判断、快速处理、完整留痕、权限明确
- 强调 7x24 值守协作
- 强调 SLA 倒计时、升级流转、告警趋势、审计可追责
- 不要做成普通 CRM、普通 OA、普通客服系统
- 不要过度互联网化，不要娱乐化，不要轻量卡片化过度

整体风格要求：
- 专业
- 冷静
- 可信
- 克制
- 偏安全运营与系统治理后台

布局基于桌面端优先，兼容：
- 1440 宽后台
- 1280 宽办公屏

界面支持：
- 大屏信息层级
- 卡片与表格混排
- 时间轴
- 状态标签
- 告警级别徽章
- 倒计时组件
- 审计日志视图
- 高密度表格
- 详情侧栏 / 非模态抽屉 / 辅助信息面板

二、多页面设计原则

请按“整套系统多页面”组织设计稿，而不是只输出一个超重的单页面。设计稿应至少体现以下层级：
- 全局导航框架
- 一级模块页面
- 核心业务详情页
- 典型状态页
- 非模态抽屉、侧栏、下载入口等局部交互

设计原则：
- 一个页面负责一个主要任务
- 详情页可以嵌入辅助信息，但不能代替其他模块的完整工作台
- 跨模块内容以引用视图、预览入口、下载入口为主
- 页面之间通过导航、面包屑、卡片入口、链接进行衔接
- 同一套组件在多页面中保持一致

三、国际化与主题

系统必须支持：
- 中文 / English 双语切换
- Light / Dark Theme 双主题切换

要求：
- 顶部导航提供语言切换与主题切换入口
- 菜单、按钮、标签、表头、状态、提示语、空状态、错误状态全部支持双语
- 预留英文文本更长的布局空间
- 暗色主题偏 SOC 夜间值守场景，避免纯黑
- 亮色主题偏白天办公与报表阅读，保持企业级克制

四、角色体系

系统包含以下角色：
- T1 分析员
- T2 分析员
- T3 专家
- 管理员
- 客户

支持当前激活角色切换。设计中需体现：
- 当前激活角色
- 可切换角色列表
- 当前角色影响的菜单可见范围与操作权限
- RBAC 权限控制
- 对象级访问控制
- 菜单可见 ≠ 数据可见 ≠ 操作可执行
- 升级流程是附加流程，不替代工单主状态

工单页权限补充规则：
- 工单状态信息对除客户之外的所有内部角色可见
- 工单被分配给指定人员时，分配关系用于展示责任归属，但不限制内部用户操作
- 工单位于池子中时，池子归属用于展示责任归属，但不限制内部用户操作
- 对内部角色来说，工单操作主要受“业务状态”约束，而不是受“当前处理人 / 当前池子”约束
- 客户仍按客户权限模型受限，只能看到其授权范围内的公开信息和允许的业务动作

五、导航结构

请设计统一后台导航结构，一级菜单包含：
1. 工作台 / Dashboard
2. 工单中心 / Tickets
3. 工单池 / Ticket Pool
4. SLA 监控 / SLA Monitor
5. 通知中心 / Notifications
6. 知识库 / Knowledge Base
7. 报告 / Reports
8. KPI / KPI
9. 配置中心 / Configuration
10. 用户、角色与权限 / Users, Roles & Permissions
11. 审计 / Audit

请同时体现不同角色的菜单差异：
- T1：工作台、工单中心、工单池、SLA 监控（受限）、通知中心、知识库、报告
- T2：工作台、工单中心、工单池、SLA 监控、通知中心、知识库、报告、KPI（受限）
- T3：工作台、工单中心、工单池、SLA 监控、通知中心、知识库、报告、KPI
- 管理员：全部菜单
- 客户：工作台、我的工单、知识库、报告、通知中心

六、页面清单

请输出以下多页面设计：

1. 登录页
- 用户名密码登录
- 安全提示
- 统一失败反馈
- 不使用验证码

2. 多角色工作台首页
- T1/T2/T3：我的待办、待响应工单、处理中工单、即将超时工单、升级待处理、通知、风险分布、告警趋势、SLA 倒计时
- 管理员：系统概览、用户与角色、配置入口、SLA 达标率、审计告警、平台运行健康
- 客户：我关注的工单、最近更新、待我回复、最新评论、最新报告

3. 工单列表页
- 高密度表格
- 左侧高级筛选
- 上方搜索栏
- 快捷筛选
- 批量操作
- 创建工单按钮
- 支持按创建时间范围筛选
- 支持固定列、排序、列管理、保存视图

列表页字段建议至少包含：
- 工单 ID
- 标题
- 分类
- 优先级
- 主状态
- 子状态
- 当前处理人
- 当前池子
- 创建时间
- 更新时间

筛选与搜索要求：
- 筛选必须支持：分类、优先级、工单状态、工单创建时间范围
- 工单状态筛选可以采用“主状态”或“主状态 + 子状态”的方式展示，但必须清晰区分
- 创建时间范围筛选必须支持开始时间与结束时间
- 搜索功能当前仅支持按工单 ID 搜索，不支持按标题、描述全文搜索

4. 工单详情页
这是整套系统中最核心的页面，但它属于多页面体系中的一个核心详情页，不是系统唯一页面。

布局要求：
- 顶部工单头部信息区
- 主内容区
- 右侧辅助信息区
- 右侧非模态抽屉

主内容区包含：
- 工单基础信息
- 主状态 / 子状态
- SLA 倒计时
- 当前责任人 / 当前池子
- 风险分
- 分类与来源
- 操作按钮区
- 评论协作区
- 操作日志时间轴
- 升级记录
- 外部上下文摘要

状态模型必须明确区分主状态与子状态：
- 主状态：WAITING_RESPONSE、IN_PROGRESS、RESPONSE_TIMEOUT、RESOLUTION_TIMEOUT、RESOLVED、CLOSED、REOPENED
- 子状态：NONE、ESCALATION_PENDING_CONFIRM、ESCALATION_CONFIRMED、ESCALATION_REJECTED

当前责任信息必须体现互斥规则：
- 工单在池中时，当前处理人为空，当前池子高亮
- 工单被领取后，当前池子为空，当前处理人高亮

责任归属与操作关系要求：
- 当前处理人和当前池子用于表达责任归属、流转关系、统计口径
- 对内部角色来说，责任归属不是操作门槛
- 即使工单已分配给指定人员，其他内部用户仍可进行允许状态下的工单操作
- 即使工单位于某个池子中，内部用户仍可进行允许状态下的工单操作

操作按钮需体现真实业务动作：
- 响应
- 处置完成
- 关闭
- 重开 Reopen
- 升级申请
- 确认升级
- 拒绝升级
- 加入池子
- 领取工单
- 编辑

操作权限要求：
- 对 T1、T2、T3、ADMIN 而言，工单操作默认不受“assigned_to 是否为自己”限制
- 对 T1、T2、T3、ADMIN 而言，工单操作默认不受“current_pool_code 是否属于自己”限制
- 详情页中可以继续展示当前处理人和当前池子，但不要把它设计成只有责任人才能操作的界面
- 动作显隐优先受业务状态控制，客户动作仍单独受限
- 升级确认与升级拒绝是例外动作，仅允许当前升级请求的目标确认人处理
- 对除目标确认人之外的其他内部用户，不显示或禁用“确认升级 / 拒绝升级”

编辑边界要求：
- 标题、描述、分类、优先级、风险分等可编辑
- 来源 source 为创建后只读字段，不应设计成可随意修改

工单详情页右侧辅助信息区必须包含“知识库卡片”和“报告卡片”，并遵循以下规则：

知识库卡片：
- 展示与当前工单分类强关联的知识文章
- 每张卡片至少包含：文章标题、文章类型、更新时间、摘要、预览入口
- 点击知识库卡片时，不跳转新页面，使用右侧抽屉进行预览
- 预览抽屉必须是“非模态抽屉”，不要用全屏遮罩锁死页面
- 抽屉打开后，原来的工单详情页仍然可以点击和操作
- 抽屉更像持续存在的侧边预览面板，而不是阻塞式弹窗
- 知识库内容默认使用 Markdown 编写
- 知识库抽屉用于直接渲染 Markdown 文件内容，并展示目录、摘要、引用关系、更新时间
- 如需查看完整文章，可在抽屉中提供“进入知识库详情页”的次级入口

报告卡片：
- 展示与当前工单分类相关的报告模板、历史报告或工单关联报告
- 每张卡片至少包含：报告标题、报告类型、状态、更新时间、下载链接
- 报告不需要预览抽屉
- 报告当前阶段不支持预览
- 点击报告时，使用链接或下载按钮直接下载
- 报告卡片必须清晰体现“下载”动作，不要设计成预览动作

工单详情页中的评论区要求：
- 客户协作主要通过评论完成，不作为独立导航模块存在
- 支持公开评论与内部协作评论区分
- 支持附件、引用、@提醒
- 支持状态变更时自动生成系统评论
- 工单关闭后仍可查看历史评论
- 客户可通过评论补充说明、反馈问题、追加上下文

页面内容以中文为主，同时能够自然切换为英文界面，适合企业内部系统与客户协作使用。

请优先输出信息架构清晰、布局稳定、可落地前端实现的企业级后台界面，不要生成概念海报式设计，不要牺牲信息密度换取装饰性。
```

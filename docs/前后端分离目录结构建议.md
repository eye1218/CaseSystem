# CaseSystem 前后端分离目录结构建议

## 现状判断

当前项目是一个单体结构：

- 后端 API 在 `casesystem/`
- 页面路由在 `casesystem/ui_routes.py`
- 前端 HTML、CSS、JS、知识库静态资源在 `casesystem/ui/static/`
- `FastAPI` 同时负责接口和页面文件分发

这个结构适合 MVP，但后续会遇到几个问题：

- 前端无法独立使用 `Vite`、`Vue`、`React` 等工程化能力
- 接口版本、跨域、鉴权、部署边界不清晰
- 测试会把“接口测试”和“页面静态资源测试”耦合在一起
- 后续如果要单独发布 Web 前端，当前目录会阻碍拆分

## 推荐目标

建议调整为“单仓库、前后端分目录”的结构，而不是直接拆成两个独立仓库。

## 为什么这样划分

### 1. `backend/` 只保留 API 和业务

后端只负责：

- 鉴权
- 权限
- 工单业务接口
- 数据库和领域规则

不再负责：

- HTML 页面返回
- 前端静态资源托管

这样可以把后端定位成标准 API 服务。

### 2. `frontend/` 作为独立 Web 工程

前端单独管理：

- 页面路由
- 组件
- 状态管理
- 接口请求封装
- 打包构建

这样你后续接入 `Vue 3 + Vite`、`React + Vite` 都很自然。

### 3. API 分层要比现在更清楚

建议后端内部再拆成几层：

- `api/`: 路由和请求入口
- `services/`: 业务编排
- `db/`: 数据库连接和模型
- `schemas/`: Pydantic 输入输出
- `core/`: 配置、安全、枚举
- `domain/`: 权限策略和领域规则

当前 `auth.py` 文件承担的职责偏多，后续最好拆到 `services/`、`domain/`、`schemas/`。

## 当前文件到新结构的迁移建议

### 后端部分

当前文件建议迁移如下：

```text
casesystem/main.py                -> backend/app/main.py
casesystem/config.py              -> backend/app/core/config.py
casesystem/security.py            -> backend/app/core/security.py
casesystem/enums.py               -> backend/app/core/enums.py
casesystem/database.py            -> backend/app/db/session.py
casesystem/models.py              -> backend/app/db/models.py
casesystem/bootstrap.py           -> backend/app/db/bootstrap.py
casesystem/schemas.py             -> backend/app/schemas/auth.py
casesystem/auth.py                -> backend/app/services/auth_service.py
casesystem/policies.py            -> backend/app/domain/policies.py
casesystem/dependencies.py        -> backend/app/api/deps.py
```

### 前端部分

当前静态页面建议不要继续保留为多份 HTML 文件，而是迁移成前端路由页面：

```text
casesystem/ui/static/tickets.html        -> frontend/src/pages/tickets/
casesystem/ui/static/ticket-detail.html  -> frontend/src/pages/ticket-detail/
casesystem/ui/static/ticket-create.html  -> frontend/src/pages/ticket-create/
casesystem/ui/static/ticket-pool.html    -> frontend/src/pages/ticket-pool/
casesystem/ui/static/recycle-bin.html    -> frontend/src/pages/recycle-bin/
casesystem/ui/static/app.js              -> frontend/src/
casesystem/ui/static/app.css             -> frontend/src/styles/
casesystem/ui/static/favicon.svg         -> frontend/public/favicon.svg
```

知识库和模板类静态资源建议按用途迁移：

```text
casesystem/ui/static/knowledge/          -> frontend/src/assets/knowledge/ 或 docs/knowledge/
casesystem/ui/static/reports/            -> docs/templates/ 或 frontend/src/assets/templates/
```

如果这些文件是“用户下载模板”，建议保留在后端对象存储或下载接口后面，不要直接混在前端源码里。

## 你这个项目更适合的最小拆分方案

如果你现在不想一次性重构太大，建议按下面顺序做。

### 第一步：先把目录拆开，不改业务逻辑

- 新建 `backend/`
- 把当前 `casesystem/` 和 `tests/` 迁进去
- 保持 Python 代码逻辑基本不变
- 先让后端仍然可以单独运行

这个阶段的目标只是“物理分层”，不是彻底重写。

### 第二步：删除后端页面路由职责

- 删除 `casesystem/ui_routes.py`
- 删除 `register_ui(app)` 挂载
- 后端只保留 `/auth/*`、`/admin/*`、`/objects/*` 这类 API
- 增加 `CORS` 配置，允许本地前端开发地址访问

这一步完成后，后端就成为真正 API 服务。

### 第三步：建立前端工程

建议前端初始化为一个标准 SPA 工程，例如：

- `frontend + Vite + Vue 3`
- 或 `frontend + Vite + React`

然后把现有几个页面逐个迁移：

- 工单列表
- 工单创建
- 工单详情
- 工单池
- 回收站

### 第四步：统一接口前缀

建议所有接口统一挂到：

```text
/api/v1/*
```

例如：

```text
/api/v1/auth/login
/api/v1/auth/logout
/api/v1/auth/me
/api/v1/admin/overview
```

这样后续做版本升级、网关转发、接口文档都更清楚。

## 当前代码里需要同步注意的点

### 1. `allowed_origins` 需要补开发环境地址

当前配置只允许：

- `https://testserver`
- `https://localhost`
- `https://127.0.0.1`

如果前端用 `Vite`，开发环境通常会是：

- `http://localhost:5173`
- `http://127.0.0.1:5173`

前后端分离后，这部分必须调整，否则浏览器会被跨域拦住。

### 2. Cookie 鉴权方案要和前端部署方式一起设计

你现在使用的是基于 Cookie 的认证和 CSRF 防护。这个方案可以继续用，但要明确：

- 前后端是否同域部署
- 是否跨子域部署
- `cookie_domain` 怎么配
- `cookie_secure` 在本地开发如何处理

如果这些没先定清楚，前端联调会反复踩坑。

### 3. 现有 UI 测试要迁移

当前：

- `tests/test_ui_pages.py`

这是在测后端返回 HTML 和静态资源。

前后端分离后应改成：

- 后端只保留 API 测试
- 前端页面测试放到 `frontend` 自己的测试体系里

## 建议你现在直接采用的版本

## 结论

对于你当前这个项目，不建议继续在 `casesystem/ui/static/` 上迭代页面。

更合理的做法是：

- `backend/` 保留 Python API 服务
- `frontend/` 独立为前端工程
- API 统一走 `/api/v1`
- 测试也分为前端测试和后端测试

这样后续无论是多人协作、部署、还是继续做工单业务，都会比现在清晰很多。

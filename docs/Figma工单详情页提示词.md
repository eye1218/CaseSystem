# Figma 工单详情页提示词

本文档整理 SOC 7x24 工单系统中“工单详情页”的 Figma 高保真设计提示词，可直接提供给 Figma AI 或其他界面生成工具使用。

## 提示词

```text
请为一个 “SOC 7x24 工单系统” 设计高保真 Web 后台中的“工单详情页”，面向安全运营中心（SOC）场景。页面服务于 7x24 值守、安全事件响应、升级流转、SLA 管控、客户反馈、审计留痕与知识复用。整体风格必须专业、冷静、可信、克制，偏安全运营中台与系统治理后台，不要做成普通 CRM、普通客服系统或互联网风控制台。

一、页面定位
这是系统最核心的业务页面，用户需要在一个页面内完成：
- 判断当前工单状态
- 判断剩余 SLA 时效
- 快速执行关键动作
- 查看协作评论与系统留痕
- 参考同类知识与报告
- 查看外部上下文摘要
- 在不离开详情页的情况下完成升级、编辑、领取、分配等操作

页面设计目标是：
- 一眼看出当前状态
- 一眼看出剩余时效
- 一眼看出责任归属
- 一眼看出关键操作
- 一眼看出协作痕迹
- 一眼看出同类知识与报告参考入口

二、布局结构
请采用桌面端优先的三栏增强布局，适配 1440 宽后台和 1280 宽办公屏。

页面结构建议如下：
1. 顶部区域
- 工单编号、标题、风险等级、优先级、主状态、子状态
- 面包屑导航
- 当前角色标识
- 中英文切换入口
- 亮色 / 暗色主题切换入口

2. 主内容区
- 工单基础信息
- 状态流转
- SLA 倒计时与超时风险
- 当前责任人 / 当前池子
- 评论协作区
- 操作日志时间轴
- 升级记录

3. 右侧辅助信息区
- 关联知识库
- 关联报告
- 外部上下文摘要（告警信息 / SIEM 研判信息）
- 权限与访问范围提示

4. 右侧抽屉
- 从页面右侧滑出
- 用于承载编辑、升级申请、确认升级、拒绝升级、加入池子、领取工单、完整上下文查看、完整知识预览、完整报告预览、审计补充信息等内容
- 保持主详情页可见，不跳转新页面，保证上下文连续

三、主内容区设计
请重点设计以下模块：

1. 工单头部信息卡
展示：
- 工单 ID（纯数字）
- 标题
- 主状态
- 子状态
- 优先级
- 风险分
- 分类
- 来源
- 创建时间
- 更新时间

要求：
- 主状态与子状态必须视觉分离
- 明确体现“升级流程是附加流程，不替代工单主状态”
- 主状态使用更强标签，子状态使用辅助标签
- 风险分与优先级要有独立视觉组件，不要与状态标签混淆

状态模型约束：
- 主状态必须完整覆盖：WAITING_RESPONSE（待响应）、IN_PROGRESS（处理中）、RESPONSE_TIMEOUT（响应超时）、RESOLUTION_TIMEOUT（处置超时）、RESOLVED（已处置）、CLOSED（已关闭）、REOPENED（已重开）
- 子状态必须完整覆盖：NONE（无）、ESCALATION_PENDING_CONFIRM（升级待确认）、ESCALATION_CONFIRMED（升级已确认）、ESCALATION_REJECTED（升级被拒绝）
- RESPONSE_TIMEOUT 状态下仍允许执行“响应”，并记录已响应时间，但主状态仍保持 RESPONSE_TIMEOUT
- RESOLUTION_TIMEOUT 状态下仍允许继续处置，处置完成后进入 RESOLVED
- REOPENED 是短暂过渡状态，后续重新回到 WAITING_RESPONSE
- 详情页必须明确区分“主状态标签”和“升级相关子状态标签”，避免用户误以为升级流程会替代工单主流程

2. SLA 信息卡
展示：
- 响应 SLA 截止时间
- 处置 SLA 截止时间
- 当前倒计时
- 是否即将超时
- 是否已响应超时
- 是否已处置超时

要求：
- 倒计时组件必须有压迫感，但不能花哨
- 即将超时用告警橙
- 超时用风险红
- 已达成用状态绿
- 在暗色主题下依然清晰可辨

3. 当前责任信息卡
展示：
- 当前处理人
- 当前池子
- 当前责任层级（T1 / T2 / T3）
- 最近一次分配 / 升级流转摘要
- 是否在池中 / 是否已被领取

展示规则：
- 工单在池中时，当前处理人应显示为空或未领取，当前池子为高亮主信息
- 工单被具体人领取后，当前池子应显示为空或已退出池子，当前处理人为高亮主信息
- 不要把“当前处理人”和“当前池子”同时设计成并列激活态，必须体现二者互斥关系
- 当前处理人和当前池子用于表达责任归属，不作为内部用户操作门槛

4. 操作按钮区
必须包含真实业务动作：
- 响应
- 处置完成
- 关闭
- 重开 Reopen
- 升级申请
- 确认升级
- 拒绝升级
- 加入池子
- 领取工单
- 编辑

要求：
- 按钮按重要程度分层
- 主按钮突出当前最推荐动作
- 次要动作放在次级按钮或更多菜单中
- 不同角色看到的动作不同
- 无权限动作可隐藏，或展示禁用态并说明原因
- 编辑工单时必须体现字段边界：标题、描述、分类、优先级、风险分等可编辑；来源 source 为创建后只读字段，不应设计成可随意修改
- 对 T1、T2、T3、ADMIN 而言，工单操作默认不受“当前处理人是否为自己”限制
- 对 T1、T2、T3、ADMIN 而言，工单操作默认不受“当前池子是否属于自己”限制
- 内部角色的动作显隐优先受业务状态控制，而不是受责任归属控制
- 升级确认与升级拒绝是例外动作，仅允许当前升级请求的目标确认人处理
- 对除目标确认人之外的其他内部用户，不显示或禁用“确认升级 / 拒绝升级”

5. 评论协作区
这是客户参与闭环的主要入口，不作为独立导航模块存在。

展示：
- 评论列表，按时间倒序
- 区分评论身份：客户 / T1 / T2 / T3 / 管理员 / 系统
- 支持公开评论与内部协作评论区分
- 支持附件、引用、@提醒
- 支持状态变更后自动生成系统评论
- 工单关闭后仍能查看历史评论
- 客户可查看公开评论、处理进展、工单状态、关联报告
- 客户可通过评论补充说明、反馈问题、追加上下文
- 满足条件时，客户可在详情页执行 Reopen

6. 操作日志时间轴
展示：
- 创建
- 响应
- 分配
- 升级申请
- 升级确认
- 升级拒绝
- 加入池子
- 领取
- 处置完成
- 关闭
- 重开
- 系统自动动作

要求：
- 时间轴强调留痕与可追责
- 每个节点包含时间、操作者、动作、结果
- 支持查看详情入口

四、右侧辅助信息区设计
右侧辅助信息区用于“辅助判断”，不是主操作区。它必须固定存在，帮助用户快速参考，而不是把所有内容塞进主区。

架构约束：
- 右侧辅助信息区展示的是知识库、报告、外部上下文、权限提示的引用视图 / 预览视图，不是这些模块的完整主工作台
- 工单详情页可以承载摘要、预览和快速入口，但不要把知识库、报告、审计、外部集成等模块的完整主操作逻辑全部揉进工单详情页
- 跨模块内容应体现“按需嵌入、快速参考、必要时跳转对应模块详情页”的设计原则

请将右侧区域明确分为三个独立模块，不可混排：

1. 知识库
展示与当前工单分类强关联的知识文章，优先匹配同类分类。
字段包括：
- 文章标题
- 文章类型
- 更新时间
- 摘要信息
- 查看入口

设计目的：
- 工单处理过程中快速参考同类知识
- 强调标准化经验复用
- 支持从详情页快速打开知识预览抽屉

2. 报告
展示与当前工单分类强关联的报告模板、历史报告或当前工单关联报告。
字段包括：
- 报告标题
- 报告类型
- 报告状态
- 最近更新时间
- 下载入口

设计目的：
- 工单处理中快速参考同类报告和模板
- 强调复盘与沉淀
- 知识库与报告必须分别成组展示，不可合并成同一列表
- 报告当前阶段不支持预览，只提供下载动作

3. 外部上下文摘要
展示：
- 告警来源
- 规则名
- 事件时间
- 资产 / 主机 / IP
- 严重级别
- SIM 研判摘要
- 外部上下文查看入口

要求：
- 右侧只展示摘要和关键字段
- 完整内容通过“大抽屉”打开
- 不要把原始日志全部堆在右侧

4. 权限与访问范围提示
展示：
- 当前激活角色
- 当前页面可见范围
- 当前评论可见范围
- 当前字段是否因对象级权限被隐藏
- 客户不可见内部升级细节、内部审计数据、内部研判备注
- 工单状态信息对除客户之外的所有内部角色可见

五、抽屉设计规则
抽屉是详情页的重要交互容器，请设计成统一系统组件，并区分三种宽度：

1. 小抽屉
适合：
- 状态说明
- 权限说明
- SLA 规则说明
- 拒绝原因查看
- 风险分解释

2. 中抽屉
适合：
- 编辑工单
- 升级申请
- 确认升级
- 拒绝升级
- 加入池子
- 领取工单
- 分配给指定人或池子

3. 大抽屉
适合：
- 完整告警详情
- 完整 SIM 研判信息
- 知识文章预览
- 审计补充记录
- 复杂上下文信息浏览

抽屉要求：
- 从右侧滑出
- 打开后主页面仍可见
- 提交成功后局部刷新，不整页跳转
- 保持上下文连续
- 支持亮色 / 暗色主题
- 支持中文 / English 双语文案长度差异
- 知识库内容默认使用 Markdown 编写，预览时直接渲染 Markdown 文件内容

六、角色差异
请在设计中体现不同角色在同一详情页上的差异：

1. T1
- 可响应、处置、评论、发起升级申请、加入池子、领取工单
- 可查看工单状态与业务信息
- 可看知识库与报告摘要

2. T2
- 可查看升级申请、确认或拒绝部分升级、分配工单、处理池子工单
- 可查看更广范围的协作信息

3. T3
- 可处理复杂升级、执行高级指派、查看专家视角信息

4. 管理员
- 可查看全部模块
- 可见更多治理类与审计入口
- 仍要体现对象级访问边界，不表现成无限权限

5. 客户
- 只能看公开评论、状态、进展、关联报告
- 可补充评论
- 满足条件时可发起 Reopen
- 不可见内部升级流转、内部协作评论、内部审计信息、内部研判细节

内部权限补充规则：
- T1、T2、T3、ADMIN 对工单的内部业务状态可见
- 工单被分配给指定人员时，其他内部用户仍可执行允许状态下的工单动作
- 工单位于池子中时，其他内部用户仍可执行允许状态下的工单动作
- 当前处理人和当前池子主要用于责任展示、统计和协作提示，不作为内部角色的唯一操作门槛
- 升级确认与升级拒绝仅允许目标确认人处理，不因为属于内部角色就自动开放给所有人

七、视觉要求
请同时输出 Light Theme 与 Dark Theme，并保持同一套设计系统。

暗色主题：
- 深蓝灰 / 石墨灰背景
- 高对比度但不纯黑
- 适合夜间值守和监控场景

亮色主题：
- 冷灰白 / 浅灰蓝背景
- 白色或浅中性色容器
- 适合白天办公与报表阅读

状态色要求：
- 待响应（WAITING_RESPONSE）
- 处理中（IN_PROGRESS）
- 响应超时（RESPONSE_TIMEOUT）
- 处置超时（RESOLUTION_TIMEOUT）
- 已处置（RESOLVED）
- 已关闭（CLOSED）
- 已重开（REOPENED）
- 升级待确认（ESCALATION_PENDING_CONFIRM）
- 升级已确认（ESCALATION_CONFIRMED）
- 升级被拒绝（ESCALATION_REJECTED）

这些状态在亮暗两套主题下都要清晰可辨。

八、中英文国际化要求
页面必须支持中文 / English 切换，并在设计中体现：
- 顶部有语言切换入口
- 菜单、按钮、状态、表头、提示、空状态、错误状态全部支持双语
- 英文文案长度通常大于中文，请预留空间
- 标签、按钮、表格列宽、卡片标题避免因英文过长而溢出
- 日期时间格式具备本地化展示能力

九、典型状态
请在详情页中体现以下典型状态：
- 正常状态
- 加载中
- 空状态
- 错误状态
- 无权限状态
- 会话失效状态
- 超时预警状态
- 响应超时状态
- 处置超时状态
- 升级申请中
- 升级待确认
- 升级被拒绝
- 工单已关闭
- 工单已重开

十、输出要求
请输出高保真企业级后台界面设计，重点展示：
- 工单详情页完整布局
- 右侧辅助信息区设计
- 右侧抽屉设计
- 评论协作区
- 日志时间轴
- 状态标签与 SLA 倒计时组件
- 不同角色下的权限差异表现
- 中英文双语表现
- 亮色 / 暗色双主题表现

优先输出可落地前端实现的稳定布局，不要生成概念海报式设计，不要为了装饰性牺牲信息密度与业务可用性。
```

# 1. 模块概述

## 1.1 模块定位

模板渲染模块是一个通用的内容模板管理与渲染基础模块，用于根据预设模板和业务上下文数据生成最终消息内容。当前本期主要服务于邮件模板与 Webhook 模板，后续可扩展到其他消息类型，但本期不开放自定义模板类型与字段结构配置入口。

该模块本身不负责消息发送，而是为上层发送模块或业务流程提供“已渲染完成的模板结果”。因此，它在系统中的角色是“模板定义 + 模板实例配置 + 模板渲染输出”的中间层能力。

## 1.2 模块目标

本模块要解决的问题有两类：

1. 将通知内容从代码中剥离，支持以模板方式配置，减少后续改动成本。
2. 统一邮件、Webhook 等不同消息类型的内容组织方式与渲染方式，避免各模块重复实现模板替换逻辑。

## 1.3 在系统中的作用

该模块位于业务模块与发送模块之间：

* 上游业务模块负责决定何时使用哪个模板，并提供渲染上下文。
* 模板渲染模块负责加载模板、校验模板字段、执行 Jinja2 渲染，并返回结构化结果。
* 下游发送模块负责基于渲染结果执行实际发送。

## 1.4 相关模块关系

本模块与以下模块存在直接关系：

* 工单模块：当前模板变量主要来源于工单字段。
* 邮件发送模块：消费邮件模板的渲染结果。
* Webhook 调用模块：消费 Webhook 模板的渲染结果。
* 后台配置/管理模块：用于模板的创建、编辑、查看、启停、预览。

# 2. 建设目标

本期建设目标如下：

1. 建立一套统一的模板模型，支持不同模板类型下的固定字段渲染。
2. 支持邮件模板与 Webhook 模板两类模板的创建、编辑、保存、查看与启停管理。
3. 支持基于 Jinja2 的模板渲染，变量来源于工单上下文。
4. 支持在创建模板时，根据模板类型自动生成对应字段编辑区。
5. 支持模板预览与渲染校验，便于在正式使用前发现变量缺失或模板错误。
6. 保持数据模型和配置模型具备后续扩展可能，但本期不开放动态扩展能力。

# 3. 功能范围

## 3.1 本期需要实现的功能

### 3.1.1 模板类型内置

系统内置以下模板类型：

* Email
* Webhook

每种模板类型包含固定字段定义，本期由系统预置，不支持用户自行新增模板类型。

### 3.1.2 模板实例管理

支持为内置模板类型创建具体模板实例，至少包括：

* 新建模板
* 编辑模板
* 查看模板
* 启用/停用模板
* 模板预览
* 模板渲染

### 3.1.3 模板字段自动渲染界面

用户在新建模板时选择模板类型后，系统根据该类型自动生成字段编辑区，不允许用户手动增删系统内置字段。

### 3.1.4 Jinja2 模板渲染

支持在允许的字段中填写 Jinja2 模板语法，并基于工单上下文渲染输出最终结果。

### 3.1.5 Email 模板支持

Email 模板固定字段：

* subject
* body

以上字段均支持 Jinja2 渲染。

### 3.1.6 Webhook 模板支持

Webhook 模板固定字段：

* url
* method
* headers
* body

其中：

* `url` 支持 Jinja2 渲染
* `method` 为固定枚举值
* `headers` 允许配置，需参与渲染
* `body` 支持 Jinja2 渲染

### 3.1.7 Webhook 特殊规则支持

当 `method = GET` 时：

* 允许填写 `body`
* 渲染模块可正常产出 `body`
* 实际发送时忽略 `body`

### 3.1.8 渲染结果结构化输出

模板渲染后，系统应按模板类型返回结构化结果，而不是仅返回单一字符串。

### 3.1.9 预留扩展能力

底层模型需考虑后续增加新模板类型和新字段定义的可能性，但本期不开放对应管理能力。

## 3.2 本期明确不实现的功能

### 3.2.1 不实现实际发送能力

本模块不负责：

* 发邮件
* 发起 Webhook 请求
* 调度发送
* 队列投递
* 重试补偿

### 3.2.2 不实现动态模板类型配置

### 3.2.3 不实现字段自由扩展入口

本期不支持为某个模板实例单独增加、删除、重命名字段。

### 3.2.4 不实现多模板引擎支持

本期模板语法固定为 Jinja2，不支持 Mustache、Liquid 等其他模板引擎。

### 3.2.5 不实现可执行脚本能力

模板只用于内容渲染，不允许演变成可执行脚本系统，不支持任意代码执行或复杂逻辑引擎。

### 3.2.6 不实现发送编排能力

不负责消息触发策略、发送条件编排、发送时机配置等能力。

### 3.2.7 不实现模板版本管理

本期不要求完整的模板版本历史、差异对比、回滚能力。

# 5. 核心概念说明

## 5.1 模板类型

指一种固定结构的模板类别，用来定义该类模板应包含哪些字段。

本期模板类型为：

* Email
* Webhook

模板类型由系统内置，不开放人工新增。

## 5.2 模板实例

指基于某个模板类型创建的具体业务模板。例如某种工单通知邮件模板、某种工单回调 Webhook 模板。

模板实例包含具体的字段内容配置。

## 5.3 模板字段

指模板类型中预定义的组成部分。字段不是自由文本片段，而是有明确业务含义的固定项。

例如：

* Email：subject、body
* Webhook：url、method、headers、body

## 5.4 字段定义

指系统对模板字段的结构化定义，包括字段标识、显示名称、是否必填、是否支持 Jinja2、是否枚举等元信息。

本期字段定义由系统内置。

## 5.5 渲染上下文

指模板渲染时可使用的数据来源。本期已明确变量主要来自工单字段。

例如模板中可引用工单相关变量进行渲染。

## 5.6 Jinja2 渲染

指模板字段内容支持使用 Jinja2 语法占位，在渲染时结合上下文输出最终字符串或结构化值。

## 5.7 模板预览

指在不触发真实发送的前提下，使用测试数据或业务上下文对模板进行一次渲染，用于检查最终展示效果与变量是否正确。

## 5.8 渲染结果

指模板执行渲染后输出的结构化内容。不同模板类型对应不同的结构化结果。

## 5.9 系统内置字段

指本期由系统预置的模板字段，用户在模板实例层仅可填写内容，不可更改字段结构。

## 5.10 扩展能力预留

指虽然本期不开放模板类型与字段扩展入口，但底层设计需考虑未来可以新增模板类型和字段定义，避免当前实现写死为固定表单。

# 6. 业务规则

## 6.1 模板类型规则

1. 本期模板类型仅支持 Email 与 Webhook。
2. 模板类型由系统内置，不允许后台新增。
3. 不同模板类型的字段集合固定，用户不可增删字段。

## 6.2 Email 模板规则

1. Email 模板固定包含 `subject` 与 `body`。
2. `subject` 为必填。
3. `body` 为必填。
4. `subject` 与 `body` 均支持 Jinja2。

## 6.3 Webhook 模板规则

1. Webhook 模板固定包含 `url`、`method`、`headers`、`body`。
2. `url` 为必填。
3. `method` 为必填。
4. `headers` 非必填。
5. `body` 非必填。
6. `url` 支持 Jinja2 渲染。
7. `method` 为枚举值，不允许自由输入。
8. `body` 支持 Jinja2 渲染。
9. `headers` 需要支持模板渲染。

## 6.4 Webhook Method 规则

1. `method` 采用固定枚举。
2. 枚举范围至少应覆盖常见 HTTP 方法，当前已明确其为枚举，但未最终明确完整枚举列表，属于待确认项。
3. 当 `method = GET` 时，允许填写 `body`，但发送时忽略 `body`。

## 6.5 URL 渲染规则

1. Webhook 的 `url` 支持 Jinja2。
2. 渲染后应得到最终可用地址。
3. 若渲染后为空或格式明显非法，应视为异常。

## 6.6 Headers 规则

1. Webhook 需要支持 `headers`。
2. `headers` 是配置项的一部分，不是发送模块自行补全的隐式字段。
3. `headers` 的具体编辑形态本轮未最终确认，当前更稳妥的推荐方案为结构化键值对形式。
4. 若采用结构化键值对，则建议仅 value 支持 Jinja2，key 默认不支持模板，以降低歧义和非法 Header 名称风险。
5. 若后续决定使用原始 JSON 文本，则需要增加语法校验与格式校验。

## 6.7 渲染规则

1. 渲染能力应按字段逐项执行，而不是只对单一文本块处理。
2. 渲染输出需保留字段边界。
3. 渲染失败时应返回明确错误，而不是静默吞掉异常。
4. 对外部发送有风险的字段，不能以隐式空值方式继续发送。

## 6.8 模板管理规则

1. 模板实例可新建、编辑、查看、启停。
2. 系统内置字段不可删除。
3. 新建模板时，选择模板类型后系统自动生成字段编辑区。
4. 编辑模板时，字段结构保持与模板类型一致，不可更改字段集合。

## 6.9 扩展性规则

1. 本期功能上不开放动态扩展。
2. 设计上必须预留模板类型与字段定义扩展空间。
3. 后续如需支持其他消息类型，不应要求重构整个模板模型。

## 6.10 潜在冲突说明

当前存在一个边界需明确：

* 渲染模块是否在 `GET` 场景下产出 `body`：当前推荐规则是“渲染模块正常产出，发送模块忽略”。
  这是当前较稳妥方案，应在研发实现中明确分层，避免渲染模块掺入发送逻辑。

## 6.11 待确认点

1. `method` 的完整枚举范围未最终锁定。
2. `headers` 的最终编辑方式未最终锁定。
3. 渲染时变量缺失的处理策略未在本轮最终确认。
4. 模板编码是否必须唯一、唯一范围如何界定，未最终确认。
5. 模板启停状态的使用规则未完全展开。

# 7. 业务流程

## 7.1 正常流程：新建模板

1. 管理员进入模板管理页面。
2. 管理员选择新建模板。
3. 输入模板基础信息。
4. 选择模板类型。
5. 系统根据模板类型自动加载该类型的字段定义。
6. 系统自动渲染字段编辑区。
7. 管理员填写各字段内容。
8. 提交保存。
9. 系统校验基础信息与字段内容。
10. 校验通过后保存模板实例。

### 覆盖点

* Email 模板应自动展示 `subject`、`body`。
* Webhook 模板应自动展示 `url`、`method`、`headers`、`body`。

---

## 7.2 正常流程：编辑模板

1. 管理员进入模板详情或编辑页。
2. 系统根据模板实例所属类型加载固定字段。
3. 管理员修改字段内容。
4. 提交保存。
5. 系统执行字段级校验。
6. 校验通过后更新模板实例。

### 关键规则

* 允许改内容，不允许改字段结构。
* 不允许删除系统内置字段。

---

## 7.3 正常流程：模板预览

1. 管理员选择某个模板。
2. 发起预览操作。
3. 系统获取测试上下文或传入的工单上下文。
4. 系统逐字段执行 Jinja2 渲染。
5. 系统返回各字段的渲染结果。
6. 管理员查看预览内容并决定是否调整模板。

### 关键规则

* 预览不触发真实发送。
* 预览必须显示字段边界，而不是合并输出。

---

## 7.4 正常流程：业务调用模板渲染

1. 上游业务模块传入模板标识及工单上下文。
2. 模板模块读取模板实例。
3. 模板模块按字段逐项渲染。
4. 模板模块返回结构化渲染结果。
5. 上游发送模块基于结果执行后续发送。

### Email 输出结果

应至少包含：

* subject
* body

### Webhook 输出结果

应至少包含：

* url
* method
* headers
* body

---

## 7.5 异常流程：模板字段缺失或未配置

1. 用户选择模板类型后未完成必填字段配置。
2. 系统在保存时执行校验。
3. 若缺少必填字段，则保存失败并返回明确错误。
4. 用户修正后重新提交。

---

## 7.6 异常流程：模板渲染失败

1. 上下文数据缺失或模板语法异常。
2. 系统执行渲染时发生失败。
3. 系统返回明确的字段级错误信息。
4. 调用方不得将失败结果当作成功结果继续发送。

---

## 7.7 异常流程：Webhook URL 渲染异常

1. Webhook 模板中的 `url` 渲染后为空、无效或明显非法。
2. 系统应视为渲染失败。
3. 调用方不得继续发送。

---

## 7.8 边界情况：Webhook GET + Body

1. 管理员配置 `method = GET`，同时填写了 `body`。
2. 模板模块允许保存该模板。
3. 业务渲染时可正常返回 `body`。
4. 实际发送时由下游发送模块忽略 `body`。

### 关键要求

该规则必须在实现中体现，不能要求用户因 GET 而删除 body 配置。

---

## 7.9 权限相关流程

### 管理员维护流程

管理员可完成模板创建、编辑、启停、预览。

### 非管理员访问流程

非管理员若无授权，不得进入模板维护操作。

### 系统调用流程

系统任务只调用渲染，不进入人工配置界面。

---

## 7.10 状态变化流程

本期至少涉及模板启停状态：

1. 新建模板后进入初始状态。
2. 启用后可被业务引用。
3. 停用后不得用于新的业务调用，具体对已绑定业务的影响需待确认。

# 8. 功能详细说明

## 8.1 模板类型选择与字段自动生成

### 功能目的

根据模板类型固定结构自动生成编辑界面，避免手动配置字段造成不一致。

### 触发方式 / 入口

管理员新建模板时选择模板类型。

### 输入信息

* 模板类型

### 系统处理逻辑

* 查询该模板类型对应的系统内置字段定义
* 按顺序自动生成字段编辑区

### 输出结果

* 展示对应字段输入区域

### 限制条件

* 不支持用户手动新增字段
* 不支持删除系统字段

### 异常处理要求

* 若模板类型无字段定义，应提示系统配置异常

---

## 8.2 Email 模板配置

### 功能目的

支持维护邮件通知内容。

### 触发方式 / 入口

管理员新建或编辑 Email 类型模板。

### 输入信息

* subject
* body

### 系统处理逻辑

* 校验必填项
* 支持 Jinja2 内容保存
* 支持后续渲染输出

### 输出结果

* 成功保存 Email 模板实例

### 限制条件

* `subject` 必填
* `body` 必填

### 异常处理要求

* 缺失任一必填字段时不得保存

---

## 8.3 Webhook 模板配置

### 功能目的

支持维护 Webhook 请求模板。

### 触发方式 / 入口

管理员新建或编辑 Webhook 类型模板。

### 输入信息

* url
* method
* headers
* body

### 系统处理逻辑

* `url` 校验必填
* `method` 校验为合法枚举
* `headers` 参与保存
* `body` 可选保存

### 输出结果

* 成功保存 Webhook 模板实例

### 限制条件

* `url` 必填
* `method` 必填
* `method` 不允许自由输入

### 异常处理要求

* `url` 为空时保存失败
* `method` 非法时保存失败
* `headers` 格式非法时保存失败

---

## 8.4 模板预览

### 功能目的

在不真实发送的情况下验证模板渲染结果是否符合预期。

### 触发方式 / 入口

管理员在模板详情或编辑界面发起预览。

### 输入信息

* 模板实例
* 测试上下文 / 工单上下文

### 系统处理逻辑

* 按字段执行 Jinja2 渲染
* 逐字段返回渲染结果或错误

### 输出结果

* 结构化预览结果

### 限制条件

* 预览不触发发送

### 异常处理要求

* 渲染失败应明确到字段级

## 8.5 模板渲染输出

### 功能目的

向业务模块提供最终可消费的渲染结果。

### 触发方式 / 入口

上游业务模块调用模板渲染能力。

### 输入信息

* 模板标识
* 工单上下文

### 系统处理逻辑

* 加载模板
* 校验模板状态
* 按字段渲染
* 生成结构化输出

### 输出结果

* Email：subject、body
* Webhook：url、method、headers、body

### 限制条件

* 停用模板不应被正常使用
* 渲染失败不得返回伪成功结果

### 异常处理要求

* 模板不存在、模板停用、变量缺失、语法错误、字段渲染失败，均需明确返回

## 8.6 模板启停管理

### 功能目的

控制模板是否可用于业务调用。

### 触发方式 / 入口

管理员在模板管理页操作。

### 输入信息

* 模板状态变更请求

### 系统处理逻辑

* 校验操作权限
* 更新模板启停状态

### 输出结果

* 状态变更成功

### 限制条件

* 非管理员不得操作

### 异常处理要求

* 无权限时拒绝操作
* 已停用模板在调用时应返回明确提示

# 9. 接口 / 数据交互要求

## 9.1 模板基础信息

模板实例至少需要具备以下基础输入信息：

* 模板名称
* 模板编码
* 模板类型
* 模板描述（可选）
* 模板状态

其中：

* 模板类型为必填
* 模板名称应为必填
* 模板编码建议为必填且唯一，最终是否强制唯一待确认
* 模板状态由系统管理

## 9.2 Email 模板字段要求

### 必填字段

* subject
* body

### 字段说明

* `subject`：字符串，支持 Jinja2
* `body`：文本内容，支持 Jinja2

## 9.3 Webhook 模板字段要求

### 必填字段

* url
* method

### 非必填字段

* headers
* body

### 字段说明

* `url`：字符串，支持 Jinja2
* `method`：枚举值，不支持自由输入
* `headers`：需支持后续扩展，格式待最终确认
* `body`：文本内容，支持 Jinja2

## 9.4 系统生成或系统控制的信息

以下信息应由系统控制，而非用户直接决定：

* 模板所属类型对应的字段结构
* 字段顺序
* 模板状态
* 模板创建与修改相关元信息
* 渲染结果中的结构边界

## 9.5 支持扩展的内容

本期仅要求模型具备未来扩展能力，重点包括：

* 模板类型可扩展
* 字段定义可扩展
* 模板字段集合可扩展
* 字段渲染能力可扩展

但本期不提供用户可操作的扩展入口。

## 9.6 格式或取值限制

### method

* 必须为枚举值
* 不允许任意字符串

### url

* 渲染后必须可作为有效请求地址使用

### headers

* 若采用键值对结构，应能校验 key/value 基本合法性
* 若采用 JSON 文本结构，应能校验格式合法性

### body

* 允许为空
* 若配置了模板语法，应参与渲染

# 10. 验收标准

## 10.1 模板类型与字段生成

1. 新建模板时，选择 Email 类型后，系统必须自动展示 `subject`、`body` 两个字段。
2. 新建模板时，选择 Webhook 类型后，系统必须自动展示 `url`、`method`、`headers`、`body` 四个字段。
3. 模板类型切换时，字段区必须同步刷新为该类型的内置字段集合。
4. 用户不能在界面中手动新增或删除系统内置字段。

## 10.2 Email 模板保存

5. Email 模板缺少 `subject` 时不得保存。
6. Email 模板缺少 `body` 时不得保存。
7. Email 模板填写完整后可成功保存。
8. 编辑 Email 模板时，仅允许修改内容，不允许修改字段结构。

## 10.3 Webhook 模板保存

9. Webhook 模板缺少 `url` 时不得保存。
10. Webhook 模板缺少 `method` 时不得保存。
11. Webhook 模板的 `method` 若不在系统枚举范围内，不得保存。
12. Webhook 模板填写合法后可成功保存。
13. Webhook 模板可保存空 `headers` 与空 `body`。

## 10.4 渲染能力

14. 模板预览时，系统必须按字段返回渲染结果。
15. Email 模板渲染结果必须包含 `subject` 与 `body`。
16. Webhook 模板渲染结果必须包含 `url`、`method`、`headers`、`body`。
17. Webhook `url` 中配置 Jinja2 变量时，渲染后应输出正确地址。
18. `body` 中配置 Jinja2 变量时，渲染后应输出正确内容。
19. `headers` 中配置模板内容时，应按既定规则完成渲染。

## 10.5 异常处理

20. 模板语法错误时，系统必须返回明确错误，不得静默成功。
21. 渲染时若关键字段失败，系统不得返回成功结果。
22. Webhook `url` 渲染后为空或明显非法时，系统必须报错。
23. 模板不存在时，调用渲染必须返回明确错误。
24. 模板停用时，调用渲染必须返回不可用提示。

## 10.6 GET 场景

25. 当 Webhook `method = GET` 且配置了 `body` 时，模板必须允许保存。
26. 当 Webhook `method = GET` 且配置了 `body` 时，模板渲染结果中可包含 `body`。
27. 下游发送侧应能基于规则忽略 GET 的 `body`，该规则需在联调时被验证。

## 10.7 权限

28. 管理员可新建、编辑、启停模板。
29. 非授权角色不得进入模板维护操作。
30. 系统调用方可调用渲染接口，但不应具备模板维护权限。

## 10.8 扩展边界

31. 本期系统仅允许使用内置模板类型。
32. 本期不得通过任何入口新增模板类型或自定义字段结构。
33. 当前实现不得写死为仅支持 Email，必须保留支持其他模板类型的结构。

# 11. 待确认问题

以下问题会直接影响研发实现，需要在开发前确认：

1. **Webhook `method` 的完整枚举范围是什么**
   当前只确定为枚举，未最终确认是否固定为 GET、POST、PUT、PATCH、DELETE，或还包含其他方法。

2. **`headers` 的最终编辑形式是什么**
   当前只确定需要支持 `headers`，但未最终确认采用：

   * 键值对结构编辑
   * 原始 JSON 文本编辑

3. **变量缺失时的渲染策略是什么**
   当前未最终确认是：

   * 严格失败
   * 输出空字符串
   * 部分字段失败允许继续
     这是高风险规则，必须确认。

4. **模板基础信息的唯一性规则是什么**
   模板编码是否强制唯一、按全局唯一还是按模板类型唯一，需明确。

5. **模板状态的完整规则是什么**
   当前只明确有启停需求，但未确认：

   * 新建后的默认状态
   * 停用模板是否允许预览
   * 停用模板是否允许编辑
   * 停用模板对已绑定业务的影响

6. **预览所用上下文来源如何确定**
   是由用户手动提供测试数据，还是选取实际工单，或由系统提供样例，当前未最终明确。

7. **Internal Message 是否纳入本期**
   历史讨论中提过内部消息，但本轮未形成明确字段结构，因此当前文档未纳入本期实现范围。如要纳入，需先明确字段模型。

# 12. 给研发的实现提醒

1. **不要把当前实现写成“邮件模板专用”**
   本期虽然只落地 Email 与 Webhook，但模型必须是“模板类型 + 字段定义 + 模板实例”的通用结构。

2. **不要把字段结构硬编码到单一页面逻辑中**
   前端界面应基于模板类型加载字段定义并自动渲染，避免后续增加类型时整体重写。

3. **不要混淆“渲染模块”和“发送模块”的职责**
   渲染模块负责生成内容；发送模块负责实际发送。
   尤其是 `GET + body` 的规则，应在分层上体现清楚。

4. **必须保留字段级错误信息**
   无论是保存校验还是渲染失败，都应能定位到具体字段，否则模板排错成本会非常高。

5. **对高风险字段不要静默兜底**
   特别是 `url`、`subject`、`method` 这类关键字段，不能在异常时自动吞错并继续向下传递。

6. **Headers 的实现不要过早锁死**
   本期虽未最终确认编辑形态，但底层应避免写成后续无法切换的结构。

7. **扩展能力要体现在模型上，不要体现在本期功能上**
   也就是：底层预留，界面不开放。不要擅自把“可扩展”实现成“本期可自由配置”。

8. **模板预览应尽早纳入实现**
   这是模板类能力能否被稳定使用的关键，不应被当成可有可无的附属功能。

9. **模板状态要在调用链路中真正生效**
   不能只在界面上显示启停，而调用时不校验状态。

10. **变量来源当前以工单字段为主，后续可能扩展**
    因此上下文处理层应考虑后续增加其他上下文字段来源，但本期不要擅自扩展业务语义。

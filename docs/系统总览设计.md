# SOC 7x24 工单系统开发文档（AI开发指导版）

## 1. 文档定位

本文档用于指导 AI 进行系统设计与开发，重点提供高层约束，包括：系统目标、业务背景、功能模块、模块依赖关系与推荐开发顺序。

说明：

* 本文档仅做高层概述。
* 每个模块的详细设计、数据结构、接口、状态机、验收规则，后续在独立模块文档中单独展开。

---

## 2. 系统目标

建设一个面向 SOC 7x24 值守场景的工单系统，用于承载从告警到处置闭环的全过程，并支持多角色协作、SLA 管理、事件自动触发、通知联动、知识沉淀和绩效统计。

系统目标如下：

1. 支持工单全生命周期管理。
2. 支持 T1/T2/T3/管理员/客户多角色协作。
3. 支持响应 SLA 与处置 SLA 的配置和跟踪。
4. 支持基于 Event 的自动化动作执行。
5. 支持知识库、报告、告警信息、SIM 研判等上下文沉淀。
6. 支持实时通知与 7x24 值守提醒。
7. 支持员工处理数量、效率和质量的 KPI 统计。

---

## 3. 业务背景

在 SOC 日常运营中，告警研判、派单、值守、升级、处置、复核、客户反馈等环节通常分散，容易出现以下问题：

* 责任归属不清晰
* 升级流程不统一
* SLA 难以持续跟踪
* 自动化联动不足
* 知识和报告难以沉淀
* 客户参与闭环能力不足
* 缺少员工处理量与处理质量统计

因此，需要构建一个以工单为核心、以 Event 为驱动、以 SLA 为约束、以通知为联动、以知识与统计为沉淀的统一系统。

## 4. 功能模块

### 4.1 身份与权限模块

负责用户认证、角色管理、RBAC 权限控制和对象级访问控制。

### 4.2 工单核心模块

负责 Ticket 主实体、工单状态机、工单列表、工单详情和基础状态流转。

### 4.3 分配与升级模块

负责指定分配、Pool 分配、自动分配、升级申请、升级确认与拒绝。

### 4.4 SLA 模块

负责响应 SLA 和处置 SLA 的计算、计时、预警与超时判定。

### 4.5 Event 与动作编排模块

负责业务事件定义、事件触发、动作绑定、动作执行、执行日志、失败重试与死信处理。

### 4.6 协作与闭环模块

负责评论、回复、客户 Reopen、关闭复核等围绕工单闭环的协作能力。

### 4.7 知识库与报告模块

负责知识库文章、Markdown 内容、报告模板、历史报告与工单报告管理。

### 4.8 通知中心模块

负责站内通知、实时推送、已读未读、通知中心和音效提醒。

### 4.9 模板与外部集成模块

负责外部 API/Webhook 动作、模板渲染、测试发送、调用日志与失败重试。

### 4.10 外部上下文展示模块

负责按需展示工单关联告警信息与 SIM 研判信息。

### 4.11 KPI 与绩效统计模块

负责员工处理数量、处理效率、SLA 达标率及基于风险分的加权统计。

### 4.12 配置中心模块

负责优先级、工单类型、SLA、升级原因字典、关闭复核策略、Event 动作绑定等系统级配置。

### 4.13 审计与可观测模块

负责审计日志、结构化日志、指标、链路追踪和关键链路监控。

## 5. 模块依赖关系

### 5.1 依赖原则

1. 工单核心模块是业务中心。
2. Event 模块是自动化中枢。
3. 通知、外部集成、KPI 尽量通过 Event 或稳定数据接口解耦。
4. 配置中心与审计模块是横切共享模块。
5. 身份与权限模块是所有业务模块的基础前置能力。

### 5.2 主要依赖关系

* 身份与权限模块：被所有业务模块依赖。
* 工单核心模块：依赖身份与权限、配置中心、审计与可观测。
* 分配与升级模块：依赖工单核心、身份与权限、配置中心、审计与可观测。
* SLA 模块：依赖工单核心、分配与升级、配置中心、审计与可观测。
* Event 与动作编排模块：依赖工单核心、分配与升级、SLA、配置中心、审计与可观测。
* 协作与闭环模块：依赖工单核心、身份与权限、配置中心、审计与可观测、Event。
* 知识库与报告模块：依赖工单核心、身份与权限、配置中心、审计与可观测。
* 通知中心模块：依赖 Event、身份与权限、配置中心、审计与可观测。
* 模板与外部集成模块：依赖 Event、身份与权限、配置中心、审计与可观测。
* 外部上下文展示模块：依赖工单核心、身份与权限、审计与可观测。
* KPI 与绩效统计模块：依赖工单核心、分配与升级、SLA、协作与闭环、Event、身份与权限、配置中心、审计与可观测。
* 配置中心模块：依赖身份与权限、审计与可观测。
* 审计与可观测模块：作为横切能力模块，被所有写操作模块共享依赖。

## 6. 推荐开发顺序

### 第 0 阶段：基础骨架

* 身份与权限模块
* 审计与可观测模块（最小版）
* 配置中心模块（最小版）

### 第 1 阶段：工单主流程

* 工单核心模块
* 分配与升级模块

### 第 2 阶段：时效与自动化骨架

* SLA 模块
* Event 与动作编排模块

### 第 3 阶段：动作执行能力

* 通知中心模块
* 模板与外部集成模块

### 第 4 阶段：协作与资产沉淀

* 协作与闭环模块
* 知识库与报告模块
* 外部上下文展示模块

### 第 5 阶段：统计与治理增强

* KPI 与绩效统计模块
* 配置中心增强
* 审计与可观测增强

---

## 7. AI 开发约束

1. AI 必须按模块依赖顺序进行设计和开发。
2. AI 不得将通知、Webhook、统计等逻辑直接写入工单核心模块。
3. 模块之间应通过稳定接口或事件协作，避免跨模块直接操作核心数据。
4. 每个模块的详细实现必须在其独立模块文档中展开。
5. 本文档优先用于高层拆分、开发排序和模块边界约束。

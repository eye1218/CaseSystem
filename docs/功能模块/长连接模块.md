# 长连接模块开发文档

## 1. 模块目标

### 1.1 功能目标

实现一个基于 **FastAPI + Socket.IO + Redis + JWT** 的长连接模块，用于在服务端与客户端之间建立实时连接，并完成以下目标：

1. 服务端能够向在线客户端实时推送业务变更。
2. 客户端收到推送后，能够根据消息更新界面状态。
3. 支持两类消息：

   * **同步消息**：用于在线实时刷新，不做离线补偿。
   * **通知消息**：用于定向通知，支持一次性离线补偿、送达确认、已读确认。
4. 支持同一用户多标签页、多客户端同时在线，且所有在线连接都能收到属于该用户的消息。
5. 支持多实例部署，连接协调与跨实例推送依赖 Redis。
6. 对工单写操作冲突采用**乐观锁**方案，避免旧数据覆盖新数据。

### 1.2 适用场景

本模块主要服务于工单系统中的实时同步场景，例如：

* 某个工单状态被更新后，其他在线客户端需要及时刷新界面。
* 某个通知仅需推送给指定用户，并要求下次登录时补推一次。
* 同一用户在多个标签页同时登录时，所有标签页均需感知消息变化。

---

## 2. 需要实现的功能

## 2.1 长连接建立与鉴权

1. 客户端登录后，通过 Socket.IO 发起连接。
2. 连接握手阶段通过 `auth` 传递 JWT。
3. 服务端校验 JWT，解析用户身份。
4. 鉴权成功后建立连接与用户的绑定关系。
5. 将用户与 socket 连接关系写入 Redis。
6. 支持连接断开后的资源清理。

---

## 2.2 在线连接管理

1. 支持同一用户多个连接同时在线。
2. 一个用户的所有在线连接都应接收属于该用户的定向消息。
3. Redis 中需维护：

   * 用户 -> socket 集合
   * socket -> 用户
   * 用户在线状态
   * 用户最后活跃时间
4. 当至少一个连接在线时，判定该用户为在线。

---

## 2.3 消息分类与推送

模块内消息必须分为两类：

### 2.3.1 同步消息

用于页面轻同步，不做离线补偿。
典型例子：

* 工单状态变化
* 工单被分配
* 工单发生变更
* 工单评论发生变化

同步消息要求：

1. 仅推送给当前在线目标连接。
2. 不进入离线补偿队列。
3. 不要求已读。
4. 默认不做 ACK。
5. 客户端收到后不直接信任消息内容，而是根据消息中的资源标识重新查询最新数据。

### 2.3.2 通知消息

用于让用户知晓某件事，支持一次性离线补偿。
典型例子：

* 某工单升级给指定用户
* 某任务指派给指定用户
* 某条系统通知仅面向指定用户

通知消息要求：

1. 支持定向发送给指定用户。
2. 发送前先写入数据库。
3. 若用户在线，则立即推送到其全部在线连接。
4. 若用户离线，则保留为待送达状态。
5. 用户下次登录并成功建立连接后，补推一次。
6. 支持送达确认与已读确认。

---

## 2.4 工单实时轻同步

1. 当工单发生变更后，服务端推送 `ticket.changed` 同步事件。
2. 同步事件只承载：

   * 工单 ID
   * 变更类型
   * 操作人
   * 发生时间
3. 客户端收到后，根据当前页面上下文决定是否刷新：

   * 工单详情页：刷新当前工单数据
   * 工单列表页：刷新列表项或重新查询列表
4. 不直接在消息中下发完整工单数据。

---

## 2.5 离线补偿

1. 仅对通知消息生效。
2. 工单状态变更类同步消息不做离线补偿。
3. 补偿策略为：

   * 用户下次登录并连接成功后补推一次。
   * 本次补推后不进行无限重试。
4. 补偿消息需有唯一标识与明确状态。

---

## 2.6 消息确认机制

仅对通知消息启用确认机制。

状态模型固定为：

* `pending`：已创建，尚未送达
* `delivered`：客户端已确认收到
* `read`：用户已读

要求：

1. 客户端收到通知消息后，立即发送 ACK。
2. 服务端收到 ACK 后，将消息状态更新为 `delivered`。
3. 用户查看通知后，客户端上报已读。
4. 服务端收到已读事件后，将状态更新为 `read`。

---

## 2.7 多端处理

1. 同一用户允许多标签页、多浏览器、多客户端同时在线。
2. 该用户的所有在线连接都应接收属于该用户的消息。
3. 通知消息状态按**用户级**记录，不按标签页拆分。
4. 任意一个标签页 ACK 成功，即可将该通知标记为 `delivered`。
5. 任意一个标签页触发已读，即可将该通知标记为 `read`。

---

## 2.8 工单并发控制

1. 工单写操作必须使用乐观锁。
2. 工单表中必须包含 `version` 字段。
3. 客户端提交更新时必须带上当前版本号。
4. 服务端更新时，必须校验请求版本号与数据库版本号是否一致。
5. 若一致，则允许更新并将版本号递增。
6. 若不一致，则拒绝更新，并返回“数据已变更，请刷新后重试”。

---

## 3. 不需要实现的功能

以下功能明确**不在本模块范围内**：

### 3.1 不实现完整消息队列系统

本模块不是通用 MQ，不需要实现：

* 无限重试
* 死信队列
* 复杂订阅模型
* 消息优先级调度

### 3.2 不实现全量离线补偿

不需要对所有消息做离线补偿。
仅通知消息需要补偿；同步消息不补偿。

### 3.3 不实现完整协同编辑锁

不实现工单编辑锁、页面占锁、抢占锁等复杂机制。
工单并发冲突统一由乐观锁处理。

### 3.4 不实现完整终端级状态系统

不需要将通知状态精确到“某个标签页是否已读”。
状态仅记录到用户级消息。

### 3.5 不实现消息内容驱动的本地真值更新

客户端不能直接以同步消息 payload 作为最终业务真值。
同步消息只用于触发重新查询。

### 3.6 不实现复杂权限计算引擎

本模块不负责定义业务权限，只负责根据上层业务系统给出的目标范围进行推送。

### 3.7 不实现历史通知中心的高级能力

本期不包含：

* 通知分组
* 通知筛选
* 通知归档
* 通知删除回收站
* 通知搜索

## 4. 概念补充说明

## 4.1 同步消息

指用于在线实时刷新 UI 的消息。
特征是：轻量、即时、不要求补偿、不要求已读。
该类消息丢失后，客户端可通过重新查询接口恢复最新状态。

## 4.2 通知消息

指需要用户知晓的业务消息。
特征是：可定向、可补偿、可确认、可已读。
该类消息需要持久化。

## 4.3 轻同步

指服务端只推送“某个资源发生变化”的通知，不推送资源完整内容。
客户端收到后再调用 HTTP API 获取最新数据。

## 4.4 离线补偿

指通知消息在用户离线时不立即送达，而是在用户下次登录并建立长连接后补推一次。
本模块不做无限次补偿。

## 4.5 ACK

指客户端收到通知消息后的送达确认。
ACK 仅代表“客户端已收到并进入处理逻辑”，不代表“用户已阅读”。

## 4.6 已读

指用户明确查看通知后的状态。
已读必须由前端基于明确用户动作触发，不可由收到消息自动推断。

## 4.7 多连接

指同一用户同时拥有多个在线 Socket 连接，例如多个标签页。
这些连接都属于同一个用户。

## 4.8 用户级状态

指通知的送达与已读状态按用户维度记录，而不是按连接维度或标签页维度记录。

## 4.9 乐观锁

指提交更新时校验资源版本是否一致。
若不一致，则拒绝旧数据提交，避免覆盖最新状态。

## 5. 数据模型

## 5.1 Socket 同步消息结构

```json
{
  "event_type": "ticket.changed",
  "message_id": "uuid",
  "scope": "broadcast",
  "target": {
    "ticket_id": 12345
  },
  "payload": {
    "ticket_id": 12345,
    "change_type": "status_changed",
    "operator_user_id": 1001,
    "occurred_at": "2026-03-11T12:00:00Z"
  }
}
```

### 字段说明

* `event_type`：事件类型，固定为 `ticket.changed`
* `message_id`：消息唯一标识
* `scope`：消息范围
* `target.ticket_id`：受影响工单
* `payload.change_type`：变化类型，例如 `status_changed`、`assigned`
* `payload.operator_user_id`：触发变更的用户
* `payload.occurred_at`：发生时间

---

## 5.2 Socket 通知消息结构

```json
{
  "event_type": "notification.created",
  "message_id": "uuid",
  "scope": "user",
  "target": {
    "user_id": 1001
  },
  "payload": {
    "notification_id": "uuid",
    "category": "ticket_upgrade",
    "title": "工单已升级",
    "content": "工单 #12345 已升级给你处理",
    "related_resource": {
      "resource_type": "ticket",
      "resource_id": 12345
    },
    "created_at": "2026-03-11T12:00:00Z",
    "requires_ack": true,
    "requires_read": true
  }
}
```

---

## 5.3 通知表

建议表名：`user_notification`

建议字段：

* `id`
* `user_id`
* `category`
* `title`
* `content`
* `related_resource_type`
* `related_resource_id`
* `status`
* `created_at`
* `delivered_at`
* `read_at`
* `expire_at`

### 状态取值

* `pending`
* `delivered`
* `read`

---

## 5.4 工单表扩展字段

工单表需增加：

* `version`

更新成功后：

* `version = version + 1`

---

## 6. Redis 结构

Redis 用于在线连接管理与跨实例消息协调，不用于替代通知数据库。

建议 key 结构如下：

### 6.1 用户到连接集合

```text
realtime:user:{user_id}:sockets
```

类型：`SET`

### 6.2 连接到用户映射

```text
realtime:socket:{socket_id}:user
```

类型：`STRING`

### 6.3 用户在线状态

```text
realtime:user:{user_id}:online
```

类型：`STRING`

### 6.4 用户最后活跃时间

```text
realtime:user:{user_id}:last_seen
```

### 6.5 未读数缓存（可选但本期建议实现）

```text
realtime:user:{user_id}:notification_unread_count
```

## 7. 业务流程

## 7.1 连接建立流程

1. 客户端登录成功后，使用 JWT 建立 Socket.IO 连接。
2. 客户端通过 `auth` 传递 token。
3. 服务端校验 JWT。
4. 若 JWT 无效，则拒绝连接。
5. 若 JWT 有效，则解析出 `user_id`。
6. 服务端建立 `user_id -> socket_id` 绑定。
7. 将绑定关系写入 Redis。
8. 将用户标记为在线。
9. 查询该用户是否存在 `pending` 通知消息。
10. 若存在，则逐条补推一次。
11. 客户端收到通知消息后回 ACK。
12. 服务端收到 ACK 后更新消息状态为 `delivered`。

### 异常情况

* token 缺失：拒绝连接
* token 无效或过期：拒绝连接
* Redis 写入失败：连接视为失败
* 补推过程中 ACK 超时：消息保持 `pending`，等待下次连接时再补推一次

---

## 7.2 连接断开流程

1. 客户端断开 Socket.IO 连接。
2. 服务端移除 `socket_id -> user_id` 映射。
3. 从用户 socket 集合中删除该连接。
4. 若该用户已无剩余在线连接，则标记用户离线。
5. 更新最后活跃时间。

### 异常情况

* Redis 清理失败：记录错误日志并进入补偿清理机制
* 非正常断线：按正常断开流程处理

---

## 7.3 工单状态轻同步流程

1. 用户 A 通过 HTTP API 提交工单操作。
2. 服务端校验业务参数。
3. 服务端读取请求中的 `version`。
4. 服务端执行带版本号条件的更新。
5. 若版本一致，则更新成功并递增版本号。
6. 若版本不一致，则返回冲突错误。
7. 更新成功后，服务端提交事务。
8. 事务成功后，发布 `ticket.changed` 事件。
9. 目标在线客户端收到事件。
10. 客户端根据 `ticket_id` 判断当前页面是否受影响。
11. 若受影响，则通过 HTTP API 拉取最新工单数据。
12. 客户端刷新 UI。

### 异常情况

* 版本冲突：返回“数据已变更，请刷新后重试”
* 事件推送失败：不影响业务提交结果
* 客户端未在线：不同步，由客户端下次查询接口获取最新状态

---

## 7.4 通知实时推送流程

1. 某业务动作触发一条需要通知指定用户的事件。
2. 服务端先向 `user_notification` 写入一条记录，状态为 `pending`。
3. 判断目标用户是否在线。
4. 若在线，则将通知推送给该用户全部在线连接。
5. 客户端收到通知后立刻回 ACK。
6. 服务端收到任意一个连接的 ACK 后，将该通知状态更新为 `delivered`。
7. 客户端更新通知中心 UI 与未读数。

### 异常情况

* 数据库写入失败：通知创建失败，不得推送
* 用户离线：不实时推送，保留 `pending`
* 客户端收到后未 ACK：维持 `pending`
* 多个标签页重复 ACK：只处理第一次有效 ACK，后续忽略

---

## 7.5 通知离线补偿流程

1. 某条通知创建时，目标用户离线。
2. 通知记录写入数据库，状态保持 `pending`。
3. 用户下次登录并成功建立 Socket.IO 连接。
4. 服务端查询该用户所有 `pending` 通知。
5. 将这些通知逐条补推到该用户当前全部在线连接。
6. 客户端收到后回 ACK。
7. 服务端收到 ACK 后更新状态为 `delivered`。
8. 本次连接中不对同一条消息进行无限重试。

### 异常情况

* 用户再次连接但 ACK 仍失败：保留 `pending`
* 用户多标签页同时连接：所有标签页都收到消息，但状态只按用户级记一次

---

## 7.6 通知已读流程

1. 客户端展示通知消息。
2. 用户执行明确查看动作，例如：

   * 打开通知中心
   * 打开消息详情
   * 点击进入关联资源页面并确认查看
3. 客户端向服务端上报已读。
4. 服务端将该消息状态更新为 `read`。
5. 服务端更新未读数。
6. 客户端刷新通知中心状态。

### 异常情况

* 已经是 `read` 的消息再次上报：幂等处理
* 多标签页同时上报已读：只处理第一次有效更新

---

## 7.7 多端并发提交流程

1. 同一工单在多个标签页同时打开。
2. 标签页 A 和标签页 B 均持有旧版本数据。
3. A 先提交更新，服务端校验版本一致，更新成功。
4. 服务端递增版本号并推送 `ticket.changed`。
5. B 随后提交旧版本数据。
6. 服务端发现版本不一致，拒绝更新。
7. B 收到冲突提示：“数据已变更，请刷新后重试”。
8. B 客户端刷新后再决定是否重新提交。

---

## 8. 验收标准

以下验收项必须基于上述业务流程执行。

## 8.1 连接与鉴权验收

1. 当客户端使用合法 JWT 建立 Socket.IO 连接时，服务端应允许连接成功。
2. 当客户端未提供 JWT 时，服务端应拒绝连接。
3. 当客户端提供无效或过期 JWT 时，服务端应拒绝连接。
4. 连接成功后，Redis 中应存在用户与 socket 的绑定关系。
5. 同一用户开启多个标签页后，Redis 中应能记录多个 socket 连接。
6. 当所有连接断开后，用户在线状态应更新为离线。

---

## 8.2 同步消息验收

1. 当工单状态成功更新后，服务端应推送 `ticket.changed` 事件。
2. `ticket.changed` 事件中必须包含：

   * `message_id`
   * `ticket_id`
   * `change_type`
   * `operator_user_id`
   * `occurred_at`
3. 客户端收到同步消息后，不得直接以 payload 替代业务真值。
4. 客户端收到同步消息后，应重新查询最新工单数据并刷新 UI。
5. 当客户端离线时，系统不得为同步消息创建离线补偿记录。
6. 同步消息推送失败时，不得影响工单业务更新结果。

---

## 8.3 通知消息验收

1. 当业务产生通知事件时，系统必须先写入通知表，再执行推送。
2. 若目标用户在线，通知应推送到该用户全部在线连接。
3. 若目标用户离线，通知应保留为 `pending`。
4. 同一用户多个标签页同时在线时，所有标签页都应收到通知。
5. 任意一个标签页 ACK 成功后，该通知状态应变为 `delivered`。
6. 重复 ACK 不得导致重复状态更新或报错。

---

## 8.4 离线补偿验收

1. 仅通知消息允许进入离线补偿流程。
2. 同步消息不得进入离线补偿流程。
3. 用户离线期间创建的通知，在下次成功连接后必须补推一次。
4. 补推后若收到 ACK，状态应更新为 `delivered`。
5. 补推后若未收到 ACK，状态应保持 `pending`。
6. 系统不得对同一条 `pending` 消息在单次连接内无限重复推送。

---

## 8.5 已读机制验收

1. 客户端收到通知后自动 ACK，但不得自动标记为已读。
2. 用户执行明确查看动作后，客户端应能上报已读。
3. 服务端收到已读上报后，应将状态从 `delivered` 或 `pending` 更新为 `read`。
4. 已是 `read` 的通知重复上报时，系统必须幂等处理。
5. 多标签页重复上报已读时，系统必须幂等处理。
6. 已读后未读数应正确更新。

---

## 8.6 乐观锁验收

1. 工单更新请求必须携带 `version`。
2. 当请求版本号与数据库版本一致时，更新应成功。
3. 更新成功后，工单版本号必须递增。
4. 当请求版本号与数据库版本不一致时，更新必须被拒绝。
5. 被拒绝时，系统必须返回明确冲突提示。
6. 冲突发生后，客户端刷新数据后应能重新提交。

---

## 8.7 多端处理验收

1. 同一用户多个标签页同时在线时，所有标签页均可收到消息。
2. 通知状态必须按用户级记录，不得按标签页拆分多条通知状态。
3. 同一条通知在多个标签页收到后，只允许第一次有效 ACK 改变状态。
4. 同一条通知在多个标签页已读后，只允许第一次有效已读改变状态。
5. 多端同时操作同一工单时，后提交的旧版本请求必须被拒绝。
6. 被拒绝的客户端应收到明确提示，并可刷新后继续操作。

---

## 9. 功能边界总结

### 9.1 本模块负责

1. 维护实时连接
2. 基于 JWT 鉴权
3. 管理用户与 socket 的绑定关系
4. 推送同步消息与通知消息
5. 实现通知消息的一次性离线补偿
6. 实现通知消息的送达确认与已读确认
7. 配合 Redis 支持多实例分发
8. 为工单并发提交提供乐观锁支持

### 9.2 本模块不负责

1. 定义工单业务规则本身
2. 定义完整权限模型
3. 实现复杂 MQ 机制
4. 实现全量历史消息中心高级功能
5. 实现编辑锁与协同占锁
6. 实现终端级已读追踪
7. 让消息 payload 直接成为业务真值

---

## 10. 给 AI 的实现约束

1. 不得将同步消息与通知消息混为一种模型。
2. 不得对所有消息统一做离线补偿。
3. 不得让前端直接信任同步消息 payload 作为最终状态。
4. 不得跳过通知表持久化直接发送通知消息。
5. 不得将送达确认与已读确认合并为一个状态。
6. 不得将通知状态细化到标签页级。
7. 不得使用编辑锁替代乐观锁。
8. 所有接口、事件、状态变更必须保证幂等处理。
9. 业务成功与消息推送成功必须解耦，推送失败不能回滚已成功的业务事务。
10. 未确认的新需求必须单独列为待确认项，不得自行扩展实现。

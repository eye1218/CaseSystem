# 1. 项目的目标

梳理并固化工单从创建进入工单池，到被领取、被管理员分配、被升级、被目标用户接受或拒绝后的完整流转规则。
输出一套边界清晰、权限明确、可直接交给研发或 AI 继续实现的工单流转需求说明，避免后续在池子归属、责任人变更、升级回退等关键规则上产生歧义。

---

# 2. 项目需实现的功能或模块

## 2.1 本期需要实现的功能

1. 工单创建后进入指定工单池，未指定时默认进入 T1 池。
2. 工单创建时记录来源，来源包括：

   * API
   * INTERNAL
   * CUSTOMER
3. 工单池内工单的查看、领取、升级能力。
4. 已被领取或已被分配工单的升级能力。
5. 管理员直接分配工单给指定用户。
6. 工单升级到上一级工单池。
7. 工单升级给指定用户，并支持目标用户接受或拒绝。
8. 升级被拒绝后的归属回退。
9. 工单在“未领取池单”和“已领取 / 已分配”两种责任态之间的切换规则。
10. 与通知、Event、审计相关的触发点定义。

## 2.2 本期功能边界

### 需要明确纳入本期的边界

1. 本期只处理 **T1 / T2 / T3 三个固定工单池** 的流转规则。
2. 本期定义 **内部用户** 的领取、升级、分配规则，并补充单租户下客户查看工单的只读规则。
3. 本期只定义 **管理员直接分配**，不实现自动分配。
4. 本期只定义 **升级到上一级池子** 和 **升级给指定用户** 两类升级方式。
5. 本期只定义升级结果，不扩展复杂审批链、会签、多人协同接单。

### 本期明确不实现的功能

1. 自动分配工单。
2. 按策略自动推荐升级对象。
3. 多级审批流、会签流、转签流。
4. 自定义工单池数量或动态池层级。
5. 客户侧参与领取、分配、升级。
6. 复杂租户隔离下的流转差异化规则。
7. 通知中心详情页、通知分组、通知搜索等高级通知能力。
8. Event 执行编排细节、任务执行细节、消息发送实现细节。
9. SLA 计时规则本身的详细设计，本期仅保留与流转相关的状态兼容要求。

# 3. 项目中涉及的概念

## 3.1 工单池

工单池是工单处于“公共待处理”状态时的归属容器。
工单在池中时，表示该工单尚未由某个具体处理人负责，而是由某个责任层级的公共队列承接。

## 3.2 当前处置人

当前处置人是当前对工单负直接处理责任的用户。
当工单被领取或被管理员分配给具体用户后，工单进入“已领取 / 已分配”状态，但仍可保留原工单池用于筛选、统计和可见性展示。

## 3.3 归属池子

指工单当前属于某个工单池，尚未落到某个具体处理人名下。
此时工单应满足：

* 有工单池归属
* 无当前处置人

## 3.4 归属个人

指工单当前已经由某个具体用户负责处理。
此时工单应满足：

* 有当前处置人
* 可以同时保留原工单池
* “是否已被领取”以当前处置人是否存在为准

## 3.5 领取

领取是指具备该池领取权限的用户，为池中工单填写当前处置人的动作。
领取成功后，该工单不能再被其他用户重复领取，且 `current_pool_code` 保持原值不变。

## 3.6 分配

分配是指管理员将工单直接指定给某个用户负责处理的动作。
分配后的归属效果与领取一致，都会填写当前处置人，但不会清空工单当前池子。

## 3.7 升级到池子

升级到池子是指将工单转移到上一级工单池继续处理。
该操作不需要目标方确认，提交后立即生效。

## 3.8 升级给指定用户

升级给指定用户是指将工单发送给某个内部合法用户，由目标用户决定接受或拒绝。
该操作需要目标用户反馈结果，不是立即生效的直接归属切换。

## 3.9 原池子

原池子是指工单在发起“升级给指定用户”之前，如果该工单仍处于“归属池子”形态时所在的池子。
当目标用户拒绝该升级请求时，系统需要将工单恢复到该原池子。

## 3.10 原处置人

原处置人是指工单在发起“升级给指定用户”之前，如果该工单已处于“归属个人”形态时负责该工单的用户。
当目标用户拒绝该升级请求时，系统需要将工单恢复给该原处置人。

# 4. 项目的上下文信息

1. **工单核心模块**
   负责工单主实体、工单状态、工单详情、工单生命周期基础能力。
   本次流转规则基于工单核心模块已有的工单对象、状态体系和工单来源字段展开。

2. **身份与权限模块**
   负责用户身份识别、角色识别、当前激活角色、权限控制。
   本次流转中的领取权限、分配权限、升级权限都依赖该模块提供的角色与权限判断。

3. **长连接 / 通知模块**
   负责向在线用户推送同步消息与通知消息。
   本次流转中的“升级给指定用户”“接受升级”“拒绝升级”“负责人变更”“池子变更”应作为通知或同步消息触发来源。

4. **Event 模块**
   负责接收业务事件并触发后续动作。
   本次流转中的工单创建、领取、分配、升级、接受、拒绝等动作需要提供稳定事件触发点，但 Event 模块不负责定义流转规则本身。

5. **系统总览依赖关系**
   当前系统整体上以工单核心为业务中心，权限模块是前置基础能力，通知和 Event 为后续联动能力。
   因此本次文档只定义工单流转本身，不将通知实现、事件编排、统计等逻辑直接耦合进流转规则中。

# 5. 业务流程

## 5.1 什么是业务流程

业务流程是指：**系统在某个业务场景中，从输入到输出的完整处理过程。**

它至少描述：

* 谁或什么触发流程
* 输入了什么
* 系统如何处理
* 最终输出什么结果

业务流程用于帮助研发理解系统在不同条件下的行为，包括正常情况和异常情况。

## 5.2 业务流程输出要求

以下流程围绕工单流转定义，已尽量覆盖：

* 正常路径
* 异常路径
* 权限限制
* 前置条件不满足
* 重复操作
* 状态异常

# 7. 使用的技术架构

1. **系统架构类型**
   采用模块化后端架构。工单流转规则由工单核心 / 分配升级相关业务服务承载，权限、通知、Event、审计通过模块边界协作。

2. **权限控制方式**
   依赖统一身份与权限模块，所有流转动作都必须基于当前激活角色进行权限判断，不允许各业务模块自行维护一套权限规则。

3. **事件驱动方式**
   工单流转成功后，输出稳定的业务事件给 Event 模块和通知模块消费，实现业务规则与后续动作解耦。

4. **实时通知方式**
   通过长连接模块向目标用户推送同步消息或通知消息；其中“升级给指定用户”“接受升级”“拒绝升级”更适合作为通知消息处理。

5. **并发控制方式**
   对领取、升级请求处理等关键流转动作采用强一致控制策略，避免同一工单被重复领取、同一升级请求被重复处理。

6. **审计方式**
   所有关键流转动作必须记录审计日志，便于追踪责任归属变化和排查异常。

# 8. 非技术约束

1. **权限约束**

   * 内部用户可查看所有工单，但不代表可执行所有流转动作。
   * 单租户模式下，客户也可查看所有工单，但不可执行领取、分配、升级等内部流转动作。
   * 领取权限、升级权限、分配权限必须分开判断。
   * 非管理员不得执行直接分配。
   * 已归属个人的工单，非当前处置人不得发起升级。

2. **审计约束**
   以下动作必须记录可追踪审计信息：

   * 创建工单
   * 领取工单
   * 分配工单
   * 升级到池子
   * 升级给指定用户
   * 接受升级
   * 拒绝升级
   * 负责人变更
   * 池子变更

3. **数据一致性约束**
   工单在任一时刻可处于以下责任态之一：

   * 未领取池单：有工单池归属、无当前处置人
   * 已领取 / 已分配：有当前处置人，且允许同时保留原工单池
     不允许工单池和当前处置人同时为空，也不允许 `assigned_to` 与 `assigned_to_user_id` 彼此不一致后继续参与流转。

4. **稳定性约束**
   对同一工单的并发领取、对同一升级请求的重复处理，必须有明确的冲突拦截机制，不能出现双负责人、双处理结果等不一致状态。

5. **范围约束**
   本文档只定义工单流转规则，不扩展自动分配、复杂审批流、通知中心高级能力、任务执行编排等超出当前目标的能力。

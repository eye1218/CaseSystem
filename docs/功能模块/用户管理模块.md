# 1. 项目的目标

建设一个仅允许管理员操作的用户管理与用户组管理模块，用于统一维护系统内全部用户（包括客户用户）的账号状态与分组关系。
该模块需要保证：停用用户无法登录、仅允许删除从未参与任何业务的用户、系统中最后一个有效 ADMIN 不可被停用或删除。

# 2. 项目需实现的功能或模块

## 2.1 本期需要实现的功能

### 用户管理

1. 新增用户
2. 查看用户列表
3. 查看用户详情
4. 编辑用户基础信息
5. 停用用户
6. 启用用户
7. 删除用户
8. 维护用户与用户组的关联关系

### 用户组管理

1. 新增用户组
2. 查看用户组列表
3. 查看用户组详情
4. 编辑用户组基础信息
5. 删除用户组
6. 向用户组添加用户
7. 从用户组移除用户

### 约束联动

1. 停用用户后，该用户不得再登录系统
2. 已登录但随后被停用的用户，其后续受保护请求应因用户状态不可用而被拒绝，重新进入登录流程
3. 删除用户前必须校验该用户从未参与任何业务
4. 最后一个有效 ADMIN 不允许被停用或删除
5. 用户可加入多个用户组
6. 用户组可用于后续批量管理、通知对象筛选、列表筛选、前端部分菜单展示过滤

## 2.2 功能边界

1. 本模块管理对象是**全部系统用户**，包括内部用户与客户用户。系统整体支持 T1 / T2 / T3 / ADMIN / CUSTOMER 多角色协作，本模块不局限于管理员账号本身。
2. 本模块中的“用户组”是业务辅助分组能力，不是权限组。
3. 用户组对“页面某些菜单可见性的过滤”仅用于**前端展示层过滤**，不参与后端鉴权。
4. 菜单、权限校验、默认工作台仍以当前激活角色为准，不能被用户组替代。

# 3. 项目中涉及的概念

| 概念       | 定义                                                                                 |
| -------- | ---------------------------------------------------------------------------------- |
| 用户       | 系统登录主体，包含内部用户与客户用户。一个用户至少有一个角色，也可以绑定多个角色。                                          |
| 有效用户     | 处于可登录、可被系统识别为正常账号状态的用户。停用用户不属于有效用户。                                                |
| 停用       | 对用户做状态标记，使其不允许登录；不是删除账号，也不是删除其历史业务关系。现有认证流程已要求校验“用户状态是否允许登录”。                      |
| 删除用户     | 对用户执行物理删除。本期仅允许删除从未参与任何业务的用户。删除后该用户账号不再存在。                                         |
| 有效 ADMIN | 当前仍处于可用状态，且拥有 ADMIN 角色的用户。系统必须始终至少保留一个有效 ADMIN。现有系统中 ADMIN 是关键管理角色，但不是绕过一切规则的超级身份。 |
| 用户组      | 用于批量管理、通知筛选、列表筛选、前端部分菜单展示过滤的分组对象，不承载后端权限。                                          |
| 菜单展示过滤   | 仅影响前端界面的显示与隐藏，不改变服务端权限校验结果。服务端权限仍以当前激活角色与对象权限为准。                                   |
| 业务参与     | 指用户已经在系统中留下业务关联，例如创建、处理、评论、关闭、升级、审计留痕、作为责任人或相关人出现在业务数据中。只要发生过业务参与，即不允许物理删除。        |

# 4. 项目的上下文信息

1. **身份与权限模块**
   负责用户认证、角色识别、角色级权限控制、对象级访问控制、登录安全与会话校验。
   本模块依赖它完成以下事项：

   * 管理接口仅允许 ADMIN 访问
   * 登录时校验用户状态是否允许登录
   * 受保护请求校验用户是否仍为 active 状态
   * 菜单与权限口径继续以当前激活角色为准，不被用户组覆盖

2. **审计与可观测模块**
   所有关键写操作模块都应接入审计能力。用户新增、编辑、停用、启用、删除、用户组变更、成员变更均属于关键写操作，应可追溯。

3. **各业务模块**
   工单、知识库、任务、协作等模块都依赖统一身份上下文。本模块删除用户时，必须校验该用户是否从未参与任何业务；否则不得删除，以避免破坏既有业务链路和审计可追溯性。

4. **系统角色模型**
   系统已有固定角色模型，当前阶段不做细粒度权限编辑器。用户组不能演变为新的权限体系。

# 5. 业务流程

## 5.1 什么是业务流程

业务流程是指：**系统在某个业务场景中，从输入到输出的完整处理过程。**
它描述谁触发、输入什么、系统如何处理、最终输出什么结果，用于指导研发理解正常、异常、权限、边界与状态变化场景。

## 5.2 业务流程输出要求

以下流程已覆盖：正常路径、异常路径、权限限制、前置条件不满足、重复操作、状态异常等情况。

# 7. 使用的技术架构

1. **架构类型**
   采用与现有系统一致的服务端统一认证与业务模块分层架构。用户管理与用户组管理作为后台管理能力模块，依赖统一身份与权限模块提供认证与鉴权。

2. **认证与鉴权方向**
   继续沿用现有 Cookie + Access Token + Refresh Token + 服务端会话的认证方案；本模块不单独实现一套登录鉴权逻辑。

3. **状态校验方向**
   用户停用后，登录流程与受保护请求流程都必须复用现有“用户状态校验”能力，不允许只在前端做限制。

4. **数据一致性方向**
   用户删除、用户组删除、成员关系变更均属于写操作，应在单次业务事务中完成核心数据变更与约束校验，避免产生半成功状态。

5. **审计方向**
   用户新增、编辑、停用、启用、删除、用户组新增/编辑/删除、成员增删均应记录审计日志；安全相关状态变化应与现有安全事件能力兼容。

6. **扩展方向**
   用户组模型当前仅作为分组对象设计，保留未来与通知筛选、前端展示过滤等能力集成的空间，但本期不扩展为权限中心。

# 8. 非技术约束

1. **权限约束**
   本模块全部写操作仅允许 ADMIN 执行。现有系统中 CUSTOMER 不可访问后台配置接口，后台管理能力应遵循相同口径。

2. **审计约束**
   关键写操作必须可追溯，不能出现无法定位操作者与操作结果的情况。

3. **安全约束**
   停用不是删除，但必须真实影响登录与受保护请求，不能仅停留在界面层面。

4. **边界约束**
   不得借“用户组”名义实现隐式权限控制；后端权限判断仍以当前激活角色、对象权限、状态机规则和责任域规则为准。

5. **数据保护约束**
   删除用户必须先校验“从未参与任何业务”，避免破坏工单、知识库、任务、审计等链路的可追溯性。

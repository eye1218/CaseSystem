# 工单升级通知弹窗 Figma 提示词

## 目标

为 CaseSystem 设计并回看“工单升级给指定用户”场景下的通知弹窗。该弹窗不是阻塞式模态，而是一个持续存在、必须响应但不阻塞页面操作的右上角通知卡片。

## 设计范围

本次只设计与工单升级相关的 3 个关键界面状态：

1. 发起方在工单详情页发起“升级给指定用户”
2. 目标用户收到待处理通知弹窗
3. 通知中心中可操作的升级通知行

## 风格约束

1. 严格沿用当前项目已有后台视觉语言：
   - 高密度
   - 专业克制
   - 表格优先
   - 右侧抽屉与局部操作面板优先
2. 不使用全屏遮罩，不做营销化弹窗，也不做夸张动效。
3. 中英文和 light/dark 都要成立。
4. 弹窗的重点是“必须回应，但不打断当前工作流”。

## 页面状态一：工单详情页内的发起操作区

页面：`/tickets/:id`

需要展示：

1. 右侧责任归属区域中新增一个紧凑操作面板
2. 该面板支持三种模式：
   - 管理员直接分配
   - 升级给指定用户
   - 升级到上一级池子
3. “升级给指定用户”模式下必须包含：
   - 目标用户选择器
   - 备注输入框
   - 提交按钮
4. 说明文案要明确告知：
   - 此操作不会立即切换负责人
   - 系统会向目标用户发送必须处理的通知
   - 需要等待目标用户接受或拒绝

## 页面状态二：目标用户收到升级通知弹窗

弹窗位置：

1. 固定在页面右上角
2. 不使用遮罩层
3. 不阻塞底层页面点击与滚动

弹窗结构：

1. 顶部状态区
   - Badge：`需要处理 / Action Required`
   - 请求发起人
   - 时间
2. 正文区
   - 标题：例如“工单 #100177 等待你确认升级”
   - 描述：说明是谁将哪张工单升级给你
   - 辅助提示：该提示在接受或拒绝前不会消失，但不会阻塞其他页面操作
3. 底部操作区
   - `打开工单 / Open Ticket`
   - `拒绝 / Reject`
   - `接受 / Accept`

交互要求：

1. 弹窗没有关闭按钮
2. 在没有接受或拒绝前，弹窗不能被手动取消
3. 接受或拒绝成功后，弹窗消失
4. 即使弹窗存在，用户依然可以继续浏览页面、切换菜单、查看工单详情

## 页面状态三：通知中心中的可操作通知

页面：`/notifications`

需要展示：

1. 升级通知在通知列表中为普通列表项，不单独跳页
2. 行内展示：
   - 标题
   - 内容
   - 状态 badge
   - `待处理 / Action Required` 标识
3. 行内操作：
   - 打开关联工单
   - 拒绝
   - 接受

## 必须捕获的 Figma 状态

至少输出以下 4 个状态：

1. 工单详情页，打开“升级给指定用户”操作面板
2. 目标用户任意工作页面上出现右上角待处理弹窗
3. 通知中心中展示带 Accept / Reject 的升级通知行
4. 目标用户处理完成后，弹窗消失、通知状态更新

## 设计注意事项

1. 弹窗是“强提醒”，但不是“阻塞模态”。
2. 视觉层级要比普通 toast 更强，但又不能抢走整个页面控制权。
3. 操作按钮需要清晰区分主次：
   - `Accept` 为主操作
   - `Reject` 为危险/次级操作
   - `Open Ticket` 为辅助导航
4. 深色模式下仍需保证高对比度与清晰层级。

# Event 模块 AI 开发提示词

你现在要为一个工单系统设计并实现 **Event 模块**。
请严格按照以下要求理解和输出，不要自行扩展业务边界，不要把未定义能力默认做成复杂规则引擎，也不要把 Event 模块与任务模块、工单模块深度耦合。

---

## 1. 模块目标

Event 模块的目标是：

1. 提供 **Event 规则配置能力**；
2. 支持管理员配置“什么工单场景会触发 Event”；
3. 支持配置 Event 的过滤条件；
4. 支持配置 Event 的触发时间规则；
5. 支持为 Event 绑定一个或多个任务模板；
6. 在规则满足时，异步触发后续任务模块执行任务。

本期的 Event 模块本质上是一个 **Event 规则定义模块**，不是 Event 运行实例管理模块，也不是任务执行管理模块。

---

## 2. 功能范围与功能边界

### 2.1 需要实现的功能

本期 Event 模块需要实现以下功能：

1. Event 规则的创建；
2. Event 规则的编辑；
3. Event 规则的删除；
4. Event 规则的启用与停用；
5. Event 规则列表查询；
6. Event 规则详情查询；
7. Event 规则绑定任务模板；
8. Event 规则配置过滤条件；
9. Event 规则配置触发时间；
10. 基于工单对象的触发点匹配 Event 规则；
11. 在 Event 命中后，异步下发绑定的任务模板 ID。

---

### 2.2 不需要实现的功能

本期 **不需要实现** 以下功能，禁止 AI 自行扩展：

1. 不实现任务模块；
2. 不实现任务执行逻辑；
3. 不实现任务执行状态管理；
4. 不实现任务执行结果管理；
5. 不实现 Event 触发记录管理；
6. 不实现 Event 运行实例管理；
7. 不实现复杂规则引擎；
8. 不支持嵌套条件组；
9. 不支持 NOT 条件；
10. 不支持跨对象规则；
11. 不支持除工单外的其他对象类型；
12. 不支持任务执行编排、重试策略、依赖关系；
13. 不支持绑定任务参数覆盖；
14. 不支持相对时间过滤；
15. 不支持软删除，本期删除为数据库真删；
16. 不支持“一个规则多个触发点”的复杂组合能力，若未特别说明，按单条规则配置单个触发点处理。

---

## 3. 极易混淆和误解的概念

以下概念极易混淆，必须严格区分，禁止 AI 自行脑补。

### 3.1 Event

本期的 Event 指的是：

```text
Event 规则定义
````

不是：

* Event 运行实例
* Event 触发记录
* 任务执行记录

---

### 3.2 Event 模块与任务模块的边界

Event 模块只负责：

1. 配置规则；
2. 判断规则是否命中；
3. 计算触发时刻；
4. 异步下发 `task_template_id` 给任务模块。

Event 模块不负责：

1. 执行任务；
2. 记录任务状态；
3. 记录任务结果；
4. 展示任务执行详情。

---

### 3.3 任务 ID 的含义

Event 绑定的任务是：

```text
task_template_id
```

不是任务实例 ID。

也就是说：

* Event 规则绑定的是任务模板；
* 真正执行时，由任务模块基于 `task_template_id` 自行生成任务实例；
* Event 模块不关心任务实例 ID。

---

### 3.4 工单类型 与 工单分类

本期中：

```text
工单类型 = 工单分类
```

这是同一个概念，不允许拆成两个字段，不允许 AI 自行设计两套字段。

---

### 3.5 创建时间过滤

本期“创建时间过滤”仅指：

```text
绝对时间范围
```

例如：

* 2026-03-01 00:00:00 ~ 2026-03-31 23:59:59

不支持：

* 最近 30 分钟
* 最近 7 天
* 已创建超过 N 分钟
* 相对时间表达

---

### 3.6 风险分数

风险分数字段定义如下：

1. 类型为整数；
2. 取值范围固定为 `0 ~ 100`；
3. 支持区间比较；
4. 不要扩展成小数，不要扩展成任意精度分值。

---

### 3.7 Event 状态

本期 Event 规则状态固定为：

* 草稿
* 启用
* 停用
* 已删除

注意：

* 这是 **规则状态**；
* 不是 Event 运行状态；
* 不要设计成“未触发 / 已触发 / 已取消”。

---

### 3.8 提前触发与延后触发

计时 Event 的触发时间统一按以下逻辑计算：

1. 所有计时 Event 都统一基于 **工单创建时间** 计算一个目标时刻；
2. 这个目标时刻定义为 **时刻 A**；
3. 提前触发 / 延后触发，都是基于 **时刻 A** 再做偏移；
4. 不要把提前/延后作用到别的时间基准上。

即：

```text
时刻A = 工单创建时间 + 目标偏移
最终触发时刻 = 时刻A - 提前时间 或 时刻A + 延后时间
```

---

## 4. 业务规则

### 4.1 Event 类型

本期 Event 只有两种类型：

1. **普通 Event**
2. **计时 Event**

---

### 4.2 工单对象触发点

工单对象支持以下触发点：

1. 工单创建
2. 工单更新
3. 工单被领取或分配
4. 工单状态变更
5. 响应超时
6. 处置超时
7. 工单关闭
8. 工单 reopen
9. 工单升级
10. 升级给指定人员被拒
11. 升级被接收

---

### 4.3 过滤字段

本期工单过滤条件支持以下字段：

1. 优先级
2. 工单类型 / 分类
3. 风险分数
4. 创建时间

不要扩展更多字段，除非明确要求。

---

### 4.4 过滤逻辑范围

过滤逻辑固定如下：

1. 同字段多值：OR
2. 不同字段之间：AND
3. 不支持嵌套条件组
4. 不支持 NOT
5. 不支持任意逻辑表达式

例如：

* 优先级 = 高 或 紧急
* 且 工单分类 = 安全事件
* 且 风险分数在 80~100
* 且 创建时间在某绝对时间范围内

---

### 4.5 命中多条规则的处理

如果一个工单同时命中多条 Event 规则，则：

```text
全部触发
```

不要做互斥，不要做优先级覆盖，不要做去重。

---

### 4.6 重复触发

本期 Event：

```text
允许重复触发
```

也就是说，同一个工单在多次满足同一条规则时，可以重复触发。

不要额外设计“防抖”“去重窗口”“幂等窗口”。

---

## 5. 业务流程

以下业务流程必须重点描述。每个流程都要包含：

1. 正常输入；
2. 正常处理逻辑；
3. 正常输出；
4. 异常输入；
5. 异常处理逻辑；
6. 异常输出结果。

---

### 5.1 流程一：创建 Event 规则

#### 正常输入

管理员提交 Event 规则配置，输入包括：

* Event 名称
* Event 类型
* 规则状态
* 触发点
* 过滤条件
* 触发时间规则
* 绑定的 `task_template_id` 列表

#### 正常处理逻辑

1. 校验基础字段是否完整；
2. 校验 Event 类型是否合法；
3. 校验触发点是否属于允许范围；
4. 校验过滤条件字段是否合法；
5. 校验风险分数是否为 `0~100` 的整数区间；
6. 校验创建时间是否为绝对时间范围；
7. 校验计时 Event 的时间配置是否合法；
8. 校验 `task_template_id` 列表格式是否合法；
9. 保存 Event 规则；
10. 保存 Event 与任务模板的绑定关系。

#### 预期输出

* 创建成功；
* 返回 `event_id`；
* 返回完整规则信息或基础摘要信息。

#### 异常输入

例如但不限于：

1. Event 名称为空；
2. Event 类型非法；
3. 触发点非法；
4. 风险分数区间越界；
5. 创建时间范围格式错误；
6. 绑定的任务模板为空或格式错误；
7. 计时规则缺少偏移配置；
8. 提前和延后配置非法。

#### 异常处理逻辑

1. 不保存任何数据；
2. 返回明确的字段级错误信息；
3. 不允许后端兜底猜测。

#### 异常输出结果

* 返回失败；
* 返回错误码；
* 返回错误原因；
* 指明具体出错字段。

---

### 5.2 流程二：编辑 Event 规则

#### 正常输入

管理员对已有 Event 规则进行修改，输入包括：

* `event_id`
* 更新后的基础信息
* 更新后的过滤条件
* 更新后的触发时间规则
* 更新后的任务模板绑定

#### 正常处理逻辑

1. 根据 `event_id` 查询 Event 规则；
2. 校验 Event 是否存在；
3. 校验 Event 是否允许编辑；
4. 替换旧规则配置；
5. 替换旧任务模板绑定关系；
6. 保存更新后的规则。

#### 预期输出

* 编辑成功；
* 返回最新规则信息。

#### 异常输入

1. `event_id` 不存在；
2. 提交字段不合法；
3. 过滤规则格式错误；
4. 时间规则错误；
5. 任务模板 ID 列表格式错误。

#### 异常处理逻辑

1. 不更新原有数据；
2. 返回明确错误信息。

#### 异常输出结果

* 返回失败；
* 返回错误码；
* 返回错误原因。

---

### 5.3 流程三：删除 Event 规则

#### 正常输入

管理员提交删除请求，输入为：

* `event_id`

#### 正常处理逻辑

1. 查询规则是否存在；
2. 若存在，则执行数据库真删；
3. 删除 Event 规则本体；
4. 删除 Event 与任务模板绑定关系。

#### 预期输出

* 删除成功。

#### 异常输入

1. `event_id` 不存在；
2. 删除过程中发生数据库异常。

#### 异常处理逻辑

1. 若规则不存在，直接返回失败；
2. 若删除失败，回滚相关操作。

#### 异常输出结果

* 返回失败；
* 返回错误码；
* 返回错误信息。

---

### 5.4 流程四：启用 / 停用 Event 规则

#### 正常输入

管理员提交启用或停用操作，输入包括：

* `event_id`
* 目标状态：启用 / 停用

#### 正常处理逻辑

1. 查询规则是否存在；
2. 校验状态流转是否合法；
3. 更新规则状态；
4. 保存状态变更。

#### 预期输出

* 状态变更成功；
* 返回最新状态。

#### 异常输入

1. `event_id` 不存在；
2. 目标状态非法；
3. 当前状态不允许切换。

#### 异常处理逻辑

1. 不更新数据；
2. 返回错误信息。

#### 异常输出结果

* 返回失败；
* 返回错误原因。

---

### 5.5 流程五：查询 Event 规则列表

#### 正常输入

管理员输入查询条件，例如：

* 名称关键字
* Event 类型
* 规则状态
* 触发点

#### 正常处理逻辑

1. 按条件查询规则列表；
2. 返回分页结果；
3. 返回基础字段与摘要字段。

#### 预期输出

* 返回规则列表；
* 返回分页信息。

#### 异常输入

1. 分页参数非法；
2. 查询条件格式错误。

#### 异常处理逻辑

1. 返回参数错误；
2. 不进行模糊容错猜测。

#### 异常输出结果

* 返回失败；
* 返回错误信息。

---

### 5.6 流程六：查询 Event 规则详情

#### 正常输入

* `event_id`

#### 正常处理逻辑

1. 查询规则本体；
2. 查询绑定任务模板；
3. 返回完整详情。

#### 预期输出

返回内容至少包括：

* 基础信息
* 触发点
* 过滤条件
* 触发时间规则
* 绑定任务模板列表

#### 异常输入

* `event_id` 不存在

#### 异常处理逻辑

* 返回不存在错误

#### 异常输出结果

* 返回失败；
* 返回错误信息。

---

### 5.7 流程七：工单事件触发 Event 匹配

#### 正常输入

系统接收到某个工单事件，例如：

* 工单创建
* 工单更新
* 工单状态变化
* 响应超时
* 处置超时
* 工单关闭
  等。

输入应包括：

* 触发点类型
* 工单当前数据快照

#### 正常处理逻辑

1. 根据触发点筛选出对应的 Event 规则；
2. 读取规则状态，仅处理“启用”状态规则；
3. 基于工单快照判断过滤条件是否命中；
4. 如果命中：

   * 普通 Event：根据规则决定立即触发或延迟触发；
   * 计时 Event：根据工单创建时间计算时刻 A，再根据提前/延后计算最终触发时刻；
5. 将绑定的 `task_template_id` 异步下发给任务模块。

#### 预期输出

* 命中规则则下发任务模板；
* 未命中则忽略；
* 多条规则命中则全部触发。

#### 异常输入

1. 工单快照缺少必要字段；
2. 触发点类型非法；
3. 规则配置本身有脏数据；
4. 任务模板绑定数据异常。

#### 异常处理逻辑

1. 记录系统错误日志；
2. 当前规则跳过；
3. 不影响其他规则继续匹配；
4. 不因单条规则异常导致整体匹配中断。

#### 异常输出结果

* 对外不一定返回用户可见结果；
* 对系统应有明确错误日志或错误事件。

---

### 5.8 流程八：普通 Event 的时间处理

#### 正常输入

* 已命中规则
* Event 类型为普通 Event
* 规则配置为立即触发或延迟触发

#### 正常处理逻辑

1. 若为立即触发，则立即异步下发任务；
2. 若为延迟触发，则根据延迟时间进行调度；
3. 延迟到达后异步下发任务。

#### 预期输出

* 任务模板被异步下发。

#### 异常输入

1. 延迟时间为空；
2. 延迟时间为负数；
3. 延迟时间格式错误。

#### 异常处理逻辑

* 当前规则不触发；
* 记录错误信息；
* 不影响其他规则。

#### 异常输出结果

* 当前规则触发失败。

---

### 5.9 流程九：计时 Event 的时间处理

#### 正常输入

* 已命中规则
* Event 类型为计时 Event
* 工单创建时间存在
* 规则配置完整

#### 正常处理逻辑

1. 以工单创建时间为基准；
2. 计算时刻 A；
3. 基于时刻 A 计算提前或延后的最终触发时刻；
4. 到达最终时刻后异步下发任务模板。

#### 预期输出

* 在正确时刻异步下发任务模板。

#### 异常输入

1. 工单创建时间缺失；
2. 时刻 A 计算参数缺失；
3. 提前/延后配置非法；
4. 最终触发时刻计算失败。

#### 异常处理逻辑

* 当前规则不触发；
* 记录错误日志；
* 不影响其他规则。

#### 异常输出结果

* 当前规则触发失败。

---

## 6. 验收标准

验收标准必须与上述业务流程一一对应，至少包含以下内容。

### 6.1 创建规则验收

1. 能成功创建合法 Event 规则；
2. 非法字段输入会被明确拦截；
3. 任务模板绑定关系能正确保存。

### 6.2 编辑规则验收

1. 能成功修改已有规则；
2. 非法修改不会污染原数据；
3. 任务模板绑定能正确替换。

### 6.3 删除规则验收

1. 删除成功后数据库中不存在该规则；
2. 关联绑定关系同步删除；
3. 不存在的规则删除时返回明确错误。

### 6.4 启停规则验收

1. 规则可在草稿、启用、停用之间进行合法切换；
2. 非法状态切换会被拦截；
3. 规则状态更新准确。

### 6.5 列表查询验收

1. 支持按条件查询；
2. 返回分页结果正确；
3. 返回字段完整。

### 6.6 详情查询验收

1. 能返回完整规则详情；
2. 能返回绑定任务模板；
3. 不存在的规则返回明确错误。

### 6.7 规则匹配验收

1. 工单事件到达后，系统能按触发点筛选规则；
2. 能按过滤条件正确判断是否命中；
3. 命中多条规则时会全部触发；
4. 允许重复触发。

### 6.8 普通 Event 触发验收

1. 立即触发规则会立即异步下发任务；
2. 延迟触发规则会在延迟后下发任务；
3. 非法延迟配置不会触发。

### 6.9 计时 Event 触发验收

1. 计时 Event 统一基于工单创建时间计算；
2. 时刻 A 计算准确；
3. 提前/延后基于时刻 A 生效；
4. 非法时间配置不会触发。

---

## 7. 技术架构

本期建议采用清晰的模块化架构，禁止设计成过度复杂的事件总线平台。

### 7.1 推荐模块划分

1. **Event Rule API**

   * 提供 Event 规则的增删改查接口；
   * 提供启用/停用接口。

2. **Event Rule Service**

   * 负责规则校验；
   * 负责规则保存；
   * 负责规则查询。

3. **Event Matcher**

   * 接收工单触发点事件；
   * 根据触发点与过滤条件判断规则是否命中。

4. **Event Scheduler**

   * 负责普通 Event 的延迟触发；
   * 负责计时 Event 的时刻 A 计算与最终调度。

5. **Task Dispatcher**

   * 负责异步下发 `task_template_id`；
   * 不等待任务结果；
   * 不管理任务状态。

6. **Persistence Layer**

   * 保存 Event 规则本体；
   * 保存 Event 与任务模板的绑定关系。

---

### 7.2 数据结构建议

至少包含两张核心表：

#### Event 规则表

建议字段：

* id
* name
* type
* status
* trigger_point
* object_type
* filter_config
* time_rule_config
* created_at
* updated_at

#### Event 任务绑定表

建议字段：

* id
* event_id
* task_template_id

不要设计任务实例表，不要设计任务结果表。

---

### 7.3 接口建议

至少提供以下接口：

1. 创建 Event 规则
2. 编辑 Event 规则
3. 删除 Event 规则
4. 启用 Event 规则
5. 停用 Event 规则
6. 查询 Event 列表
7. 查询 Event 详情
8. 绑定任务模板
9. 查询已绑定任务模板

如果采用一次性保存完整规则，也可以把任务绑定包含在创建 / 编辑接口内，但最终仍然只绑定 `task_template_id`。

---

### 7.4 异步要求

任务下发必须满足：

1. 异步执行；
2. 不阻塞 Event 规则处理主链路；
3. 不等待任务结果；
4. 单条规则异常不影响其他规则；
5. 单个任务模板下发异常不应导致整个系统阻塞。

---

## 8. 非技术约束

### 8.1 边界约束

1. 不要越过 Event 模块边界；
2. 不要把任务模块能力做进 Event 模块；
3. 不要把工单业务逻辑硬编码进 Event 模块。

### 8.2 实现约束

1. 不要实现复杂规则引擎；
2. 不要实现多层嵌套条件；
3. 不要实现 NOT / XOR / 任意表达式；
4. 不要实现多对象混合触发。

### 8.3 术语约束

输出中必须统一术语：

* Event：指 Event 规则
* 触发点：指工单业务事件来源
* 过滤条件：指工单字段筛选规则
* 任务模板：指 `task_template_id`

不要混用：

* Event 实例
* Event 日志
* Task 实例
* Task 结果

### 8.4 输出要求

在后续开发设计、接口设计、数据库设计、前端页面设计中，必须始终遵守本文档约束。
如果某个点未明确，优先采用最简单、最保守、最不越界的实现，不要自行扩展业务能力。

---

## 9. 最终开发原则

请始终遵守以下原则：

1. **简单优先**：先实现可用的 Event 规则配置，不做平台化过度设计；
2. **边界清晰**：Event 只管规则与触发，不管任务执行；
3. **规则明确**：过滤规则只支持固定逻辑范围；
4. **异步触发**：任务必须异步下发；
5. **低猜测**：未明确的地方不要擅自扩展；
6. **强约束实现**：所有实现必须贴合本文档，不允许自由发挥。

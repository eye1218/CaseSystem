# 任务模块开发文档

## 1. 项目的目标

任务模块负责承接 Event 模块触发后的动作执行，基于任务模板生成任务实例，完成 Email / Webhook 两类任务的执行、状态管理、结果记录、执行日志记录与失败后人工重试。它的目标是把“事件触发”与“动作执行”解耦，避免 Event 模块直接承担执行细节，同时为后续扩展飞书等任务类型预留统一执行框架。

---

## 2. 项目需实现的功能或模块

### 2.1 本期必须实现

1. 任务模板管理
   支持独立维护任务模板，任务模板不是 Event 的内嵌配置，而是被 Event 通过 `task_template_id` 引用的独立实体。

2. 任务类型支持
   本期仅支持两类任务模板与任务实例：

   * Email Task
   * Webhook Task
     同时底层结构需预留后续扩展能力，避免写死为仅支持两种类型。模板模块本期同样只支持 Email / Webhook 两类模板。

3. 任务模板与模板模块的引用关系

   * Email Task 必须引用模板模块中的 `email_template_id`
   * Webhook Task 必须引用模板模块中的 `webhook_template_id`
     任务模板本身不直接保存最终发送内容，只保存执行定义与模板引用关系。模板模块负责渲染，任务模块负责执行。

4. 任务模板字段管理
   任务模板至少需要支持以下信息：

   * 名称
   * 任务类型
   * 引用模板 ID
   * 收件人 / 目标对象配置
   * 抄送 / 密送配置（Email Task）
   * 启停状态

5. 收件人来源规则
   Email Task 收件人必须支持多值，并支持以下来源：

   * 自定义邮箱
   * 系统内部当前处理人
   * 某角色下全部人员

6. Event 触发后的任务实例生成
   每当一个 Event 触发后，其绑定的每个 `task_template_id` 都必须独立生成一条任务实例；多个任务实例互不影响，其中一个失败不得阻塞其他任务实例。Event 模块本身只负责绑定任务模板和触发，不管理执行状态和结果。

7. 任务实例执行

   * Email Task：基于 Email 模板渲染结果执行邮件发送
   * Webhook Task：基于 Webhook 模板渲染结果执行 HTTP 调用
     Webhook Task 不允许在任务模板层覆盖 URL、method、headers、body，统一使用模板模块渲染结果。模板模块已明确 Webhook 输出结构为 `url / method / headers / body`，且 GET 请求场景允许保留 body 配置，但发送时下游应忽略 body。

8. 任务实例状态管理
   状态枚举固定为：

   * `PENDING`
   * `RUNNING`
   * `SUCCESS`
   * `FAILED`
   * `CANCELLED`
     其中 `CANCELLED` 本期仅作为状态预留，不进入本期实际业务流转。

9. 执行结果管理
   记录任务实例的执行结果，包括成功、失败、失败原因、外部响应摘要等。

10. 执行日志管理
    支持查看任务执行日志，至少记录：

* 任务实例 ID
* 来源 Event ID
* 任务类型
* 开始时间 / 结束时间
* 输入快照摘要
* 渲染结果摘要
* 第三方响应摘要
* 错误信息
* 操作人（如为人工重试）

11. 失败后人工重试
    本期不做自动重试；失败后在任务管理中提供“重试”按钮。点击重试时必须新建一条新的任务实例，保留原失败实例；可重试权限为“管理员 + 有该工单访问权限的人”。

12. 权限控制

* 管理员可维护任务模板、查看任务、重试任务
* 普通用户不能维护任务模板
* 仅管理员和对关联工单有访问权限的人可查看相关任务实例并执行重试
  系统整体权限体系建立在身份与权限模块及对象级访问控制之上。

13. 审计与可观测接入
    任务模板维护、任务实例创建、执行、失败、重试等关键写操作应接入审计与可观测体系。系统总览明确所有写操作模块应依赖审计与可观测能力。

### 2.2 本期明确不需要实现

1. 自动重试
2. 失败弹窗提醒
3. 取消任务实例
4. 飞书任务类型
5. 短信、站内信等其他任务类型
6. 任务模板版本管理
7. 模板结构自定义能力
8. 在任务模板中覆盖 Webhook 模板内容
9. Event 模块内直接执行任务
10. 任务模块自行承担模板渲染逻辑

### 2.3 功能边界说明

* Event 模块负责触发，不负责任务执行状态和结果。
* 模板模块负责模板维护与渲染，不负责真实发送、调度、重试。
* 任务模块负责实例生成、执行、状态、结果、日志、人工重试。
* 工单模块不直接发送通知或调用 Webhook，这类动作应通过 Event / 任务链路解耦。

## 3. 项目中涉及的概念

### 3.1 任务模板

任务模板是任务模块中的独立配置实体，用于定义“某类任务应该如何执行”。它不是最终发送内容本身，而是对模板引用、目标对象、启停状态等执行规则的封装。Event 通过 `task_template_id` 绑定任务模板，而不是直接绑定模板模块中的 Email / Webhook 模板。

### 3.2 任务实例

任务实例是某次真实执行的运行记录。它由 Event 触发后基于任务模板生成，也可以由失败任务人工重试时再次生成。任务实例是状态、结果、日志的承载对象。

### 3.3 Email Task

一种任务类型。执行时必须引用模板模块中的 Email 模板，渲染结果至少包含 `subject` 和 `body`，任务模块据此执行真实邮件发送。

### 3.4 Webhook Task

一种任务类型。执行时必须引用模板模块中的 Webhook 模板，渲染结果至少包含 `url`、`method`、`headers`、`body`，任务模块据此执行真实 HTTP 调用。任务模板层不允许覆盖这些字段。

### 3.5 模板引用

指任务模板与模板模块中具体模板实例之间的关联关系。

* Email Task → `email_template_id`
* Webhook Task → `webhook_template_id`

### 3.6 来源 Event

指触发任务实例生成的上游 Event。一个 Event 触发后，其绑定的每个任务模板独立生成一条任务实例。

### 3.7 手动重试

指任务实例执行失败后，由有权限的用户在任务管理中点击“重试”，系统新建一条新的任务实例再次执行；原失败实例保留，不被覆盖。

### 3.8 任务状态

本期统一使用以下状态：

* `PENDING`：待执行
* `RUNNING`：执行中
* `SUCCESS`：执行成功
* `FAILED`：执行失败
* `CANCELLED`：已取消（本期只预留，不实际流转）

## 4. 项目的上下文信息

### 4.1 与当前模块直接相关的上游与下游关系

* Event 模块：负责业务事件定义、状态管理、触发调度、绑定 `task_template_id`；触发后调用任务模块，但不管理任务执行状态和结果。
* 模板模块：负责 Email / Webhook 模板维护与渲染，返回结构化渲染结果；不负责真实发送与重试。
* 工单核心模块：提供工单上下文、当前处理人、对象访问权限边界，并产生与工单生命周期相关的触发点，如工单创建、关闭、Reopen、超时、升级等。
* 身份与权限模块：提供 RBAC 与对象级访问控制，是任务模板管理、任务查看、任务重试权限判断的前置模块。
* 审计与可观测模块：承接任务模块的写操作审计与执行链路观测。

### 4.2 当前模块在链路中的位置

工单 / SLA / 升级等业务动作
→ 产生 Event
→ Event 命中绑定的 `task_template_id`
→ 任务模块生成任务实例
→ 任务模块调用模板模块渲染
→ 任务模块执行 Email 发送或 Webhook 调用
→ 记录状态、结果、日志
→ 失败后允许人工重试

### 4.3 与系统总览的关系说明

系统总览中将“Event 与动作编排模块”“模板与外部集成模块”“通知中心模块”拆为解耦模块，并明确通知、Webhook 等能力不应直接写入工单核心模块。任务模块属于对“动作执行能力”进一步落地的独立执行层，用于把 Event 与外部动作执行隔离开。

## 7. 使用的技术架构

1. 模块化服务架构
   任务模块应作为独立业务模块存在，与 Event 模块、模板模块、工单模块解耦，遵循系统总览中“工单为中心、Event 为自动化中枢、通知与外部集成通过稳定接口解耦”的原则。

2. 事件驱动 + 异步执行
   Event 触发任务实例生成后，任务执行应采用异步处理，避免阻塞工单主流程。Event 模块文档已明确任务执行异步并行。

3. 模板渲染与执行分离
   模板模块只负责渲染结构化结果，任务模块消费渲染结果执行真实发送 / 调用，不在任务模块重复实现模板解析逻辑。

4. 可扩展任务类型框架
   虽然本期只实现 Email / Webhook，但设计上应保留新增其他任务类型的能力，避免流程、状态、执行器等结构写死。模板模块也已要求底层保留扩展能力。

5. 审计与可观测接入
   任务模板维护、任务执行、失败、重试等关键操作应纳入审计日志、结构化日志和关键链路监控。

6. 权限与对象访问控制
   权限判断应依赖统一的身份与权限模块，并结合工单对象访问权限控制任务查看与重试能力。

---

## 8. 非技术约束

1. 权限约束
   任务模板维护仅允许管理员；任务查看与失败重试需同时满足管理员身份或关联工单访问权限。

2. 审计约束
   所有关键写操作必须可追溯，包括任务模板变更、任务实例生成、执行结果变更、失败记录、人工重试。

3. 业务一致性约束
   一个 Event 绑定多个任务模板时，各任务实例必须独立处理，失败互不影响。

4. 扩展约束
   本期不得扩展到 Email / Webhook 之外的真实任务类型，但底层必须预留未来扩展能力。

5. 边界约束
   本期不支持自动重试、不支持失败弹窗提醒、不支持取消任务实例、不支持模板版本管理。

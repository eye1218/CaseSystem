# 报告模块设计文档

## 1. 背景与目标

### 1.1 背景

当前系统已经具备工单模块、模板渲染模块和前端工单详情页，但“报告”能力仍处于临时实现阶段：

1. 工单详情页已有“关联报告”展示区，但数据来自静态 `REPORT_LIBRARY`，不是真实业务数据。
2. `/reports` 路由目前仍是占位页，尚未形成独立模块。
3. 现有 `templates` 模块用于 Jinja2 内容渲染，不适合直接承载“可下载文件模板”这一需求。
4. 当前后端尚未建立正式的文件上传、替换、删除与权限校验链路。

本次设计的目标是把报告能力从静态占位扩展为正式模块，并与现有工单体系形成清晰边界。

### 1.2 模块目标

本模块需要达成以下目标：

1. 支持按工单类型维护多个可下载的报告模板文件。
2. 支持按工单 ID 管理用户上传的具体报告文件。
3. 支持从报告模块和工单详情页两个入口上传实际报告。
4. 支持已上传报告的列表视图与卡片视图切换。
5. 支持报告下载、元信息编辑、文件替换、硬删除。
6. 保持“报告模板”和“已上传报告”两类对象分层清晰，避免在数据结构和页面职责上混用。

### 1.3 设计原则

1. 配置能力与业务数据分离。
2. 优先复用现有工单、权限、详情聚合能力，不重做一套平行流程。
3. 数据模型以当前已确认需求为准，不提前引入审批流、在线生成、版本流转等未被要求的能力。
4. 模块边界清晰，便于后续写实现计划和拆任务。

## 2. 模块定位与范围

### 2.1 模块定位

报告相关能力分为两部分：

1. 报告模板
   - 本质是“按工单类型配置的模板文件”。
   - 用途是供内部人员下载参考，在线下填写或整理后再回传最终报告。
   - 不属于 `/reports` 主入口的业务数据，归配置侧管理。

2. 已上传报告
   - 本质是“用户上传的、与具体工单绑定的实际报告文件”。
   - 用途是沉淀工单处理过程中的交付物、总结材料、复盘材料等。
   - 属于报告模块主入口管理对象。

### 2.2 本期范围

本期需要实现：

1. 报告模板按工单类型关联。
2. 一个工单类型支持多个报告模板。
3. 模板文件支持下载。
4. 已上传报告按工单 ID 关联。
5. 已上传报告支持从工单详情页上传。
6. 已上传报告支持从报告模块页上传，上传时必须显式选择工单 ID。
7. 已上传报告支持列表展示与卡片展示。
8. 已上传报告支持下载。
9. 已上传报告支持编辑元信息。
10. 已上传报告支持替换文件。
11. 已上传报告支持硬删除。
12. 已上传报告可选关联来源模板，但不是必填。

### 2.3 本期不实现

1. 在线报告编辑器。
2. 在线模板生成报告。
3. 模板驱动的自动出报告流程。
4. 审批、签署、盖章、流转状态机。
5. 删除恢复与回收站。
6. 报告版本树、版本比较、回滚。
7. 客户上传报告。
8. 分片上传与超大文件传输。

## 3. 与现有模块的关系

### 3.1 工单模块

报告模块与工单模块直接关联：

1. 报告模板通过工单类型关联到工单。
2. 已上传报告通过工单 ID 关联到工单。
3. 工单详情页需要同时展示模板下载区和已上传报告区。
4. 报告上传、替换、删除等关键动作需要写入工单操作日志，保证可审计。

### 3.2 模板渲染模块

现有模板渲染模块不承接本次“文件模板”需求，原因如下：

1. 它的核心对象是结构化内容模板，不是文件资源。
2. 它面向的是渲染输出，不是下载文件管理。
3. 强行复用会让“内容模板”和“文件模板”语义混杂。

因此，本次设计新增独立的“报告模板”能力，不复用 `templates` 模块的数据模型。

### 3.3 权限与身份模块

报告模块沿用现有认证与角色体系，不重新定义登录态。权限规则在本模块内落地为资源访问控制。

## 4. 角色与权限

### 4.1 角色分类

1. 内部人员
   - 包括 `T1`、`T2`、`T3`、`ADMIN` 等内部角色。
2. 客户
   - 只能访问自己有权限查看的工单。
3. 配置维护人员
   - 本期明确为 `ADMIN` 角色，负责维护报告模板。

### 4.2 权限规则

#### 4.2.1 报告模板

1. 内部人员可下载与工单类型匹配的模板文件。
2. `ADMIN` 可创建、编辑、替换、启停报告模板。
3. 客户不直接接触报告模板配置入口。

#### 4.2.2 已上传报告

1. 内部人员可在自己有权限访问的工单上上传报告。
2. 内部人员可编辑自己有权限访问的报告元信息。
3. 内部人员可替换自己有权限访问的报告文件。
4. 内部人员可删除自己有权限访问的报告。
5. 内部人员可下载自己有权限访问的报告。
6. 客户不可上传报告。
7. 客户可查看并下载自己有权限访问的工单下的已上传报告。

### 4.3 访问范围约束

1. 所有报告访问都必须先通过工单访问权限校验。
2. 当客户访问报告时，必须保证该报告所属工单也在客户可见范围内。
3. 报告模块主页面默认展示当前用户有权限看到的全部已上传报告。
4. 报告模块主页面提供“我上传的”快速筛选，但不是默认唯一视图。

## 5. 核心概念与业务对象

### 5.1 报告模板 `ReportTemplate`

用于描述“某一类工单可下载的报告模板文件”。

核心特征：

1. 与工单类型绑定，不与具体工单绑定。
2. 一个工单类型可有多个模板。
3. 模板文件可被下载，但不会直接变成实际报告。
4. 模板可启用或停用。

### 5.2 已上传报告 `TicketReport`

用于描述“某个具体工单下的实际报告文件”。

核心特征：

1. 与具体工单 ID 绑定。
2. 由内部人员上传。
3. 可以可选关联一个来源模板。
4. 支持下载、编辑元信息、替换文件、硬删除。

### 5.3 工单类型

本设计中的“工单类型”沿用现有工单字段 `category_id` / `category_name` 的业务语义。报告模板通过 `ticket_category_id` 关联该类型。

### 5.4 来源模板

已上传报告可选记录 `source_template_id`，用于表达“这份报告是否基于某个模板文件填写而来”。该字段仅用于回溯，不参与权限和展示主逻辑。

## 6. 数据模型设计

### 6.1 报告模板表

采用新表：`report_templates`

字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 主键，UUID |
| name | string | 模板名称 |
| description | text nullable | 模板说明 |
| ticket_category_id | string | 关联工单类型 ID |
| status | string | `ACTIVE` / `INACTIVE` |
| original_filename | string | 原始文件名 |
| content_type | string nullable | 文件 MIME 类型 |
| size_bytes | int | 文件大小 |
| storage_key | string | 物理存储键 |
| created_by_user_id | string nullable | 创建人 |
| updated_by_user_id | string nullable | 更新人 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

设计说明：

1. 模板文件本身存储为文件资源，不落入 JSON 字段。
2. `ticket_category_id` 是模板查找主条件。
3. 模板不引入删除恢复能力，本期通过启停控制可见性与可用性。

### 6.2 已上传报告表

采用新表：`ticket_reports`

字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 主键，UUID |
| ticket_id | int | 关联工单 ID |
| title | string | 报告标题 |
| report_type | string | 报告类型，如结案报告、复盘报告、阶段报告 |
| note | text nullable | 备注或说明 |
| source_template_id | string nullable | 来源模板 ID，可为空 |
| original_filename | string | 原始文件名 |
| content_type | string nullable | 文件 MIME 类型 |
| size_bytes | int | 文件大小 |
| storage_key | string | 物理存储键 |
| uploaded_by_user_id | string nullable | 上传人用户 ID |
| uploaded_by_name | string | 上传人显示名 |
| created_at | datetime | 首次上传时间 |
| updated_at | datetime | 最近编辑或替换时间 |

设计说明：

1. `ticket_id` 是业务主关联，不可为空。
2. `title`、`report_type`、`note` 构成最小业务元信息集合。
3. `source_template_id` 可空，但若填写，必须能校验到对应模板存在且与工单类型匹配。
4. 因为删除方式为硬删除，不增加 `is_deleted`、`deleted_at` 等软删除字段。

### 6.3 文件存储约束

本期不引入独立对象存储抽象平台，先采用后端统一文件存储能力承载，要求如下：

1. 业务表只保存文件元信息和 `storage_key`。
2. 下载必须通过后端鉴权接口完成，不直接暴露物理路径。
3. 替换文件时必须生成新的物理存储键，避免覆盖导致缓存脏读。
4. 删除报告时必须同时删除数据库记录与物理文件。
5. 删除失败时应阻止数据库提交，避免出现“记录没了但文件还在”或“文件没了但记录还在”的半成功状态。

### 6.4 文件类型约束

本期采用固定上传格式白名单，规则如下：

1. 允许上传 `pdf`、`doc`、`docx`、`xls`、`xlsx`、`ppt`、`pptx`、`txt`、`md`。
2. 禁止可执行文件类型与压缩包类型。
3. 单文件大小上限为 `50MB`。
4. 不支持压缩包内解析和内容预览。
5. 下载时按原始文件名回传。

## 7. 功能设计

### 7.1 报告模板管理

报告模板属于配置侧能力，不在 `/reports` 主入口展示。

本期需要支持：

1. 新建模板
   - 上传模板文件
   - 填写模板名称
   - 选择工单类型
   - 填写模板说明
2. 编辑模板元信息
   - 修改名称
   - 修改说明
   - 调整启停状态
3. 替换模板文件
4. 下载模板文件
5. 按工单类型查询模板列表

模板业务规则：

1. 一个工单类型可配置多个模板。
2. 同一模板必须只属于一个工单类型。
3. 停用模板不再出现在工单详情的模板下载区。
4. 已上传报告若引用了某模板，即使模板后来停用，已上传报告上的来源模板信息仍可保留。

### 7.2 已上传报告管理

本期需要支持：

1. 上传报告
2. 查看报告列表
3. 卡片/列表视图切换
4. 下载报告
5. 编辑元信息
6. 替换文件
7. 删除报告

#### 7.2.1 上传报告

支持两个入口：

1. 工单详情页上传
   - 当前工单 ID 自动带入
   - 用户不可改关联工单
2. 报告模块页上传
   - 必须显式选择工单 ID
   - 只有在工单权限校验通过后才允许提交

上传时需要填写或提供：

1. 关联工单 ID
2. 报告标题
3. 报告类型
4. 备注/说明，可为空
5. 来源模板，可为空
6. 上传文件

#### 7.2.2 编辑元信息

允许修改：

1. 标题
2. 报告类型
3. 备注
4. 来源模板

不允许修改：

1. 所属工单 ID
2. 上传人
3. 首次上传时间

#### 7.2.3 替换文件

替换文件的语义是“保留同一条报告记录，更新其承载文件”。

替换后需要更新：

1. `original_filename`
2. `content_type`
3. `size_bytes`
4. `storage_key`
5. `updated_at`

替换后不改变：

1. 报告 ID
2. 所属工单 ID
3. 首次上传时间
4. 上传人

#### 7.2.4 删除报告

删除语义为硬删除。

删除时必须：

1. 校验当前用户对所属工单有内部操作权限。
2. 删除数据库记录。
3. 删除物理文件。
4. 写入工单操作日志。

本期不提供恢复入口。

### 7.3 报告模块主页面

`/reports` 页面只展示“已上传报告”。

主页面要求：

1. 默认展示当前用户有权限看到的全部已上传报告。
2. 提供“我上传的”快速筛选。
3. 支持列表视图与卡片视图切换，且针对同一数据集切换。
4. 支持从该页面发起上传。

基础筛选固定为：

1. 关键字搜索，匹配标题。
2. 工单 ID。
3. 报告类型。
4. 我上传的。

基础排序固定为：

1. 按上传时间倒序，作为默认排序。
2. 按最近更新时间倒序，可作为补充排序。

### 7.4 工单详情页集成

工单详情页需要拆成两个业务区块：

1. 报告模板下载区
2. 已上传报告区

#### 7.4.1 报告模板下载区

规则如下：

1. 根据当前工单 `category_id` 查询有效模板列表。
2. 仅展示 `ACTIVE` 状态模板。
3. 每条模板提供下载入口。
4. 当该工单类型没有模板时，展示空状态。

#### 7.4.2 已上传报告区

规则如下：

1. 仅展示当前工单下的已上传报告。
2. 内部人员可在此直接上传。
3. 客户仅可查看和下载。
4. 每条报告展示标题、类型、上传时间、上传人和下载入口。
5. 内部人员可在此执行编辑、替换文件、删除。

## 8. 关键业务规则

### 8.1 模板与工单类型匹配规则

1. 报告模板必须绑定到一个确定的工单类型。
2. 工单详情页只拉取当前工单类型下的模板。
3. 上传报告时，如果填写了来源模板，该模板必须属于当前工单的工单类型。
4. 如果来源模板与工单类型不匹配，接口必须拒绝请求。

### 8.2 报告与工单绑定规则

1. 已上传报告必须绑定到一个存在且当前用户可访问的工单。
2. 报告一旦创建，`ticket_id` 不允许修改。
3. 报告模块页上传时，必须在提交前完成工单存在性与权限校验。

### 8.3 显示规则

1. 模板不出现在 `/reports` 主列表。
2. `/reports` 只展示已上传报告。
3. 工单详情页同时展示模板和已上传报告，但两个区域必须分开。

### 8.4 审计规则

以下动作必须写入工单操作日志：

1. 上传报告
2. 编辑报告元信息
3. 替换报告文件
4. 删除报告

日志内容至少应包含：

1. 操作类型
2. 报告标题
3. 操作人
4. 操作时间

### 8.5 可见性规则

1. 已上传报告不额外定义独立可见级别，沿用所属工单的可见性。
2. 客户只能看到自己有权限访问的工单下的报告。
3. 内部人员只要能访问工单，就能访问工单下报告。

## 9. 后端模块边界

### 9.1 新增后端单元

后端新增两个子模块：

1. `backend/app/modules/report_templates`
   - 负责报告模板的模型、校验、服务、路由。
2. `backend/app/modules/reports`
   - 负责已上传报告的模型、校验、服务、路由。

### 9.2 与 tickets 模块的接口边界

`tickets` 模块不负责实现报告存储逻辑，只负责在详情聚合时调用报告模块输出。

边界如下：

1. `tickets` 负责工单存在性与访问权限的基础判断。
2. `reports` 负责报告资源本身的增删改查。
3. `report_templates` 负责按工单类型返回模板列表。
4. `tickets` 详情接口负责把模板列表和已上传报告列表整合进详情响应。

### 9.3 文件存储边界

文件存储能力放在独立的轻量服务层，不直接散落在路由中，避免后续替换存储实现时影响业务层。

## 10. 接口设计

### 10.1 报告模板接口

提供以下接口：

1. `GET /api/v1/report-templates`
   - 查询模板列表
   - 支持按 `ticket_category_id`、`status` 过滤
2. `POST /api/v1/report-templates`
   - 创建模板并上传文件
3. `GET /api/v1/report-templates/{template_id}`
   - 查询模板详情
4. `PATCH /api/v1/report-templates/{template_id}`
   - 更新模板名称、说明、状态
5. `POST /api/v1/report-templates/{template_id}/replace-file`
   - 替换模板文件
6. `GET /api/v1/report-templates/{template_id}/download`
   - 下载模板文件

### 10.2 已上传报告接口

提供以下接口：

1. `GET /api/v1/reports`
   - 查询当前用户可见的已上传报告列表
   - 支持关键字、工单 ID、报告类型、我上传的筛选
2. `POST /api/v1/reports`
   - 上传新报告
   - 使用 `multipart/form-data`
3. `GET /api/v1/reports/{report_id}`
   - 查询报告详情
4. `PATCH /api/v1/reports/{report_id}`
   - 更新标题、类型、备注、来源模板
5. `POST /api/v1/reports/{report_id}/replace-file`
   - 替换报告文件
6. `GET /api/v1/reports/{report_id}/download`
   - 下载报告原文件
7. `DELETE /api/v1/reports/{report_id}`
   - 硬删除报告

### 10.3 工单详情聚合接口

为了减少前端改动，在现有工单详情响应中新增或调整以下字段：

1. `report_templates`
   - 当前工单类型下可下载的模板列表
2. `reports`
   - 当前工单下真实已上传报告列表

要求：

1. 保留 `reports` 字段名，用真实数据替换当前静态数据来源。
2. 新增 `report_templates` 字段，避免前端把模板和实际报告混成一个数组。

### 10.4 上传请求格式

上传类接口统一采用 `multipart/form-data`，元信息与文件一起传输。

必须项：

1. `ticket_id`
2. `title`
3. `report_type`
4. `file`

可选项：

1. `note`
2. `source_template_id`

## 14. 数据流说明

### 14.1 模板下载流

1. 用户打开工单详情页。
2. 前端请求工单详情。
3. 后端根据工单 `category_id` 聚合有效模板列表。
4. 用户点击模板下载。
5. 后端校验访问权限后返回模板文件。

### 14.2 报告上传流

1. 内部人员在工单详情页或 `/reports` 发起上传。
2. 前端提交 `multipart/form-data`。
3. 后端校验工单存在性与访问权限。
4. 若填写来源模板，校验模板与工单类型匹配。
5. 后端保存物理文件。
6. 后端写入 `ticket_reports` 记录。
7. 后端写入工单操作日志。
8. 前端刷新当前列表。

### 14.3 报告替换流

1. 内部人员发起替换文件。
2. 后端校验权限与资源存在。
3. 保存新文件。
4. 更新报告记录文件元信息。
5. 删除旧文件。
6. 写入工单操作日志。

### 14.4 报告删除流

1. 内部人员发起删除。
2. 后端校验权限。
3. 删除物理文件。
4. 删除数据库记录。
5. 写入工单操作日志。
6. 前端从列表中移除该记录。

---

## 15. 安全与审计要求

1. 所有下载必须经过后端鉴权，不暴露裸存储路径。
2. 上传接口必须复用现有认证与 CSRF 机制。
3. 文件名展示可使用原始文件名，但物理存储名不能直接使用原始文件名。
4. 删除、替换等关键操作必须记录日志。
5. 任何跨工单越权访问都必须在后端被拦截，不能依赖前端隐藏按钮实现权限控制。

## 17. 迁移与落地要求

### 17.1 对现有静态报告实现的替换

当前 `tickets` 模块中的静态 `REPORT_LIBRARY` 仅作为临时原型数据来源，本期上线后应由真实数据库数据替代。

替换要求：

1. 工单详情中的 `reports` 字段保留，但数据源改为 `ticket_reports`。
2. 静态 `REPORT_LIBRARY` 不再作为正式业务数据来源。
3. 新增 `report_templates` 字段承接模板下载区。

### 17.2 对现有前端占位页的替换

当前 `/reports` 占位页需要替换为真实页面，且不得再以“仅在工单详情下载”作为模块说明。

## 18. 实施边界与拆分

为便于后续写实现计划，本设计拆成以下可独立理解的单元：

1. 报告模板后端模块
2. 已上传报告后端模块
3. 文件存储服务层
4. 工单详情接口聚合改造
5. 报告模块前端页面
6. 工单详情页报告区改造
7. 配置中心模板入口
8. 前后端测试补齐

每个单元职责单一，接口边界明确，可独立测试。

---

## 19. 最终结论

本次报告模块建设采用“双实体、双入口、单主模块”方案：

1. 报告模板是配置对象，按工单类型关联，归配置侧管理。
2. 已上传报告是业务对象，按工单 ID 关联，归 `/reports` 主模块管理。
3. 工单详情页同时展示模板下载区和已上传报告区。
4. 内部人员负责上传与维护，客户仅查看和下载自己工单下的报告。

该方案与现有项目结构兼容，能平滑替换当前静态原型实现，并且不会污染已有内容模板模块的职责边界。

# 身份与权限模块

## 1. 模块功能说明与目标

### 1.1 模块定位

身份与权限模块是整个系统的基础模块，负责：

* 用户登录与登出
* 会话建立与失效
* Access Token / Refresh Token 的签发、刷新、校验、撤销
* 用户角色识别
* 接口级权限控制
* 对象级访问控制
* 登录安全防护
* 为审计模块提供操作者身份

本模块不承载工单业务本身，但所有业务模块都必须依赖它提供的认证与授权能力。

### 1.2 模块目标

本模块需要达成以下目标：

1. 建立安全、可撤销、可审计的登录会话机制
2. 支持角色与权限控制，确保不同角色只能访问允许的功能
3. 支持对象级权限校验，避免越权访问工单、报告等业务对象
4. 在**不使用验证码**的前提下，通过限流、锁定、CSRF、防会话滥用等方式降低暴力破解和会话滥用风险
5. 为后续模块提供统一的身份上下文，不允许各模块自行实现一套认证逻辑

## 2. 用户角色说明

系统内置以下角色：

### 2.1 T1

一线处理人员。主要职责是接单、确认、响应、处置基础工单，可向 T2 发起升级。

### 2.2 T2

二线处理人员。具备 T1 能力，并处理升级工单，可向 T3 发起升级。

### 2.3 T3

资深处理人员。处理复杂工单和高级升级工单。

### 2.4 ADMIN

管理员。负责系统配置、权限管理、模板配置、Webhook 配置、审计查看等后台能力。

### 2.5 CUSTOMER

客户。可查看授权范围内的工单状态与报告，可评论、可对已关闭工单执行 Reopen，但无后台配置权限。

### 2.6 角色设计规则

* 一个用户至少有一个角色
* 一个用户**可以绑定多个角色**（例如同时拥有 T2 与 ADMIN）
* 系统登录后需要确定**当前激活角色（active role）**
* 用户可以通过前端菜单进行角色切换
* 权限判断以**当前激活角色**为准，而不是所有角色并集
* 角色集合用于授权范围，当前激活角色用于执行操作
* ADMIN 不是“绕过一切规则”的超级身份，仍需通过认证，且对象级访问规则应显式定义

### 2.7 当前激活角色（Active Role）

系统允许用户拥有多个角色，但同一时间只会有一个**当前激活角色**。

设计原则：

1. 用户登录后系统必须确定当前激活角色
2. 默认角色优先级：

   * 用户配置的默认角色
   * `user_roles.is_primary = true`
   * 若都不存在则按角色排序选择第一个
3. 用户可以在前端菜单中执行角色切换
4. 角色切换只影响当前会话
5. 不同浏览器会话可以拥有不同当前角色
6. 菜单、权限校验、默认工作台均以当前激活角色为准

当前激活角色必须保存在服务端会话中，而不是仅依赖前端状态。

## 3. 认证与授权总体方案

本系统采用：

* **短期 Access Token**
* **长期 Refresh Token**
* **Refresh Rotation**
* **Cookie 承载**
* **服务端会话状态**
* **无验证码**
* **三层限流 + 临时锁定**
* **CSRF 防护**

### 3.1 Access Token

Access Token 使用 JWT，负责携带短期访问凭证。JWT 的通用定义来自 RFC 7519，安全使用建议遵循 JWT BCP（RFC 8725），包括固定算法白名单、校验 `iss` / `aud` / `exp` / `nbf` / `iat` 等。

### 3.2 Refresh Token

Refresh Token 使用**不透明随机串**，不使用 JWT。
刷新时必须轮换，旧 Refresh Token 再次使用视为 reuse，整条 token family 失效。OAuth 2.0 Security BCP 明确推荐 refresh token rotation 作为重放检测手段之一。

### 3.3 Cookie 承载

敏感 Cookie 必须使用 `Secure` 和 `HttpOnly`；`__Host-` 前缀要求 `Path=/` 且不能设置 `Domain`；`__Secure-` 适合其他安全 Cookie。

### 3.4 CSRF 防护

由于认证凭证通过 Cookie 自动发送，所有状态变更接口必须做 CSRF 防护。`SameSite` 只能作为防御增强，不能替代 CSRF Token；有状态系统推荐 synchronizer token pattern。

### 3.5 无验证码策略

本模块**不使用验证码，也不接入第三方验证码服务**。
登录安全依赖：

* 账户+IP 限流
* 账户全局锁定
* IP 全局限流
* 指数退避
* 统一失败响应
* 安全日志与审计

## 4. 业务流程（闭环）

### 4.1 登录流程

#### 目标

建立一条完整、安全、可审计的登录闭环。

#### 流程

1. 用户提交 `username` 和 `password`
2. 服务端检查：

   * IP 是否被限流
   * 账户+IP 是否被限流
   * 账户是否临时锁定
   * 用户状态是否允许登录
3. 查询用户
4. 校验密码哈希
5. 若失败：

   * 更新失败计数
   * 执行指数退避
   * 判断是否触发账户+IP 限制、账户临时锁定、IP 限流
   * 记录安全事件
   * 返回统一失败响应
6. 若成功：

   * 清理账户相关失败计数
   * 创建会话 `auth_session`
   * 生成 Access Token
   * 生成 Refresh Token
   * 保存 Refresh Token 哈希
   * 设置 Cookie
   * 记录登录成功审计与安全日志
   * 返回成功

#### 成功结果

* 客户端获得合法会话
* 服务端有可追踪会话记录
* 后续请求可识别当前用户

#### 失败结果

* 不暴露“用户名不存在 / 密码错误 / 账户锁定”的具体原因
* 只返回统一失败结果
* 所有失败都要落安全日志

### 4.2 Access Token 验证流程

#### 目标

对每一个受保护请求完成身份识别。

#### 流程

1. 从 Cookie 读取 Access Token
2. 校验 JWT 格式
3. 使用固定算法白名单校验签名
4. 校验 `typ`
5. 校验 `iss`
6. 校验 `aud`
7. 校验 `exp`
8. 校验 `nbf`
9. 校验 `iat`
10. 校验 `sub` 对应用户是否存在
11. 校验用户状态是否为 `active`
12. 校验 `sid` 对应会话是否为 `active`
13. 校验 `tv` 是否等于当前用户 `token_version`
14. 校验 `rv` 是否等于当前用户 `role_version`
15. 校验通过后，将当前用户上下文注入请求上下文

#### 验证成功处理

* 放行到业务接口
* 业务层按当前用户上下文继续做权限判断

#### 验证失败处理

按失败原因分流，但对前端保持有限暴露：

* token 缺失：未认证
* token 签名错误：未认证
* token 过期：未认证
* session 已撤销：未认证
* user disabled：未认证
* token_version 不匹配：未认证
* role_version 不匹配：未认证

#### 返回策略

* 返回 401
* 前端应清理本地登录态展示并跳转登录页
* 服务端记录安全日志

### 4.3 Refresh Token 刷新流程

#### 目标

在用户无需重新输入密码的情况下延续安全会话。

#### 流程

1. 客户端调用 `POST /auth/refresh`
2. 服务端从 Refresh Cookie 读取 token
3. 对 token 做哈希，查询 `refresh_tokens`
4. 校验：

   * token 存在
   * 未过期
   * 未撤销
   * session 仍为 active
5. 若发现该 token 已被标记为 `rotated` 或已用过：

   * 认定为 refresh token reuse
   * 撤销整条 `token_family`
   * 撤销关联 session
   * 清空认证 Cookie
   * 记录安全事件
   * 返回重新登录
6. 若校验通过：

   * 生成新 Access Token
   * 生成新 Refresh Token
   * 将旧 Refresh Token 标记为 `rotated`
   * 插入新 Refresh Token 记录
   * 更新 Cookie
   * 返回刷新成功

#### 成功结果

* 会话继续有效
* Refresh Token 完成轮换
* 旧 token 立即失效

#### 失败结果

* 任何异常都不继续签发新 token
* reuse 事件要升级为高危安全事件

### 4.4 登出流程

#### 流程

1. 用户调用 `POST /auth/logout`
2. 服务端根据当前会话找到 `auth_session`
3. 撤销该 session 下所有 active refresh tokens
4. 将 session 状态标记为 `revoked`
5. 清空 Access Cookie、Refresh Cookie、CSRF Cookie
6. 写审计日志和安全日志
7. 返回成功

#### 结果

* 当前会话不可再刷新
* 旧 Access Token 在过期前可能仍存在短窗口，但由于 session 已撤销，后续校验时应拒绝

### 4.5 修改密码后的闭环

#### 流程

1. 用户修改密码成功
2. 生成新的 password hash
3. `token_version + 1`
4. 撤销该用户所有 active session
5. 撤销该用户所有 active refresh tokens
6. 写安全日志
7. 要求所有设备重新登录

#### 结果

* 所有历史 token 立即失效
* 防止旧会话继续存活

### 4.6 权限校验流程

#### 流程

1. 请求通过认证后进入业务接口
2. 根据 session 查询 `user_role_sessions.active_role_code`
3. 将该角色作为当前激活角色
4. 先执行角色级权限判断
5. 再执行对象级权限判断
6. 全部通过才允许执行业务动作
7. 写操作必须把当前用户身份与当前角色传递给审计模块

#### 角色级权限示例

* ADMIN 才能访问配置中心
* CUSTOMER 不能访问后台配置接口
* T1 不能执行 T2/T3 专属审批动作

#### 对象级权限示例

* CUSTOMER 只能访问授权范围内工单与报告
* 内部人员按系统定义访问内部范围对象
* 评论、报告下载、工单详情均要做对象级校验

#### 关键原则

权限判断 = **当前激活角色权限 × 对象权限 × 状态机规则 × 责任域规则**

## 5. 验证成功与失败的处理规则

### 5.1 登录成功

* 清理账户相关失败计数
* 更新 `last_login_at`
* 创建 session
* 签发 Access / Refresh Token
* 设置 Cookie
* 写安全日志：`LOGIN_SUCCESS`

### 5.2 用户不存在

* 不向前端暴露
* 记日志：`USER_NOT_FOUND`
* 增加 IP 失败计数
* 返回统一错误

### 5.3 密码错误

* 不向前端暴露
* 记日志：`PASSWORD_MISMATCH`
* 增加账户+IP、账户、IP 失败计数
* 触发指数退避
* 返回统一错误

### 5.4 账户临时锁定

* 前端仍返回统一失败文案
* 记日志：`ACCOUNT_TEMP_LOCKED`
* 不执行密码校验

### 5.5 用户被禁用

* 前端返回统一失败或未认证
* 记日志：`USER_DISABLED`
* 不允许建立新会话

### 5.6 Access Token 过期

* 返回 401
* 前端尝试 refresh
* refresh 失败则跳转登录

### 5.7 Refresh Token 过期

* 返回 401
* 清理 Cookie
* 强制重新登录

### 5.8 Refresh Token reuse

* 撤销整个 family
* 撤销对应 session
* 返回重新登录
* 记高危安全日志：`REFRESH_TOKEN_REUSED`

### 5.9 权限不足

* 返回 403
* 写审计日志
* 不暴露额外内部信息

## 6. 密码存储方案

### 6.1 原则

密码必须**哈希存储，不允许加密存储，不允许明文存储**。

### 6.2 固定方案

* 算法：**Argon2id**
* 最低参数：

  * memory cost：**19 MiB**
  * iterations：**2**
  * parallelism：**1**

### 6.3 实现要求

* 密码哈希由成熟库完成，不手写实现
* salt 由库自动生成
* 数据库只存哈希结果
* 可选增加 pepper，但 pepper 不入库，放到独立安全配置中
* 未来参数升级时，用户登录成功后可做透明 rehash

### 6.4 不允许的做法

* 明文密码
* 可逆加密存储密码
* 自己拼接 SHA256/SHA1/MD5
* 固定 salt
* 手工自定义哈希格式

## 7. 登录失败限流规则

### 7.1 设计原则

不使用验证码，因此必须依赖限流和锁定。

### 7.2 三层限流

#### 第一层：账户 + IP

* 窗口：10 分钟
* 阈值：5 次失败
* 处罚：该账户+IP 组合禁止继续登录 15 分钟

#### 第二层：账户全局

* 窗口：15 分钟
* 阈值：10 次失败
* 处罚：账户临时锁定 30 分钟

#### 第三层：IP 全局

* 窗口：5 分钟
* 阈值 1：20 次失败
  处理：额外固定延迟 3~5 秒
* 阈值 2：50 次失败
  处理：登录接口返回 429，封禁 1 小时
* 阈值 3：200 次失败 / 24 小时
  处理：推送到反向代理或 WAF 黑名单，封禁 24 小时

### 7.3 指数退避

从同一账户+IP 维度看：

* 第 3 次失败：延迟 1 秒
* 第 4 次失败：延迟 2 秒
* 第 5 次失败：延迟 4 秒
* 第 6 次失败：延迟 8 秒
* 第 7 次及以上：延迟上限 30 秒

### 7.4 成功登录后

* 清空该账户全局失败计数
* 清空该账户+IP 失败计数
* 保留 IP 全局统计用于安全分析

### 7.5 统一失败响应

登录失败统一返回：

**“用户名或密码错误，或当前登录暂不可用”**

不能区分：

* 用户不存在
* 密码错误
* 账户锁定
* 账户禁用

## 8. Cookie 与 CSRF 方案

### 8.1 Cookie 设计

#### Access Token Cookie

* 名称：`__Host-access_token`
* 属性：

  * `Secure`
  * `HttpOnly`
  * `SameSite=Strict`
  * `Path=/`
  * 不设置 `Domain`

#### Refresh Token Cookie

* 名称：`__Secure-refresh_token`
* 属性：

  * `Secure`
  * `HttpOnly`
  * `SameSite=Strict`
  * `Path=/auth/refresh`

#### CSRF Token Cookie

* 名称：`XSRF-TOKEN`
* 属性：

  * `Secure`
  * `SameSite=Strict`
  * `Path=/`
  * 不设置 `HttpOnly`

### 8.2 CSRF 校验规则

所有状态变更请求必须校验：

* `X-CSRF-Token`
* `Origin` / `Referer`

包括：

* `/auth/login`
* `/auth/refresh`
* `/auth/logout`
* 所有 `POST / PUT / PATCH / DELETE`

## 9. 后端数据库表设计

以下为 MVP 必需表。

### 9.1 用户表 `users`

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(64) NOT NULL UNIQUE,
    email VARCHAR(255) UNIQUE,
    display_name VARCHAR(128) NOT NULL,
    password_hash TEXT NOT NULL,
    password_algo VARCHAR(32) NOT NULL DEFAULT 'argon2id',
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    token_version INTEGER NOT NULL DEFAULT 1,
    role_version INTEGER NOT NULL DEFAULT 1,
    lock_until TIMESTAMPTZ NULL,
    last_login_at TIMESTAMPTZ NULL,
    password_changed_at TIMESTAMPTZ NULL,
    disabled_at TIMESTAMPTZ NULL,
    disabled_reason TEXT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by UUID NULL,
    updated_by UUID NULL
);
```

#### 字段说明

* `id`：用户主键，固定 UUID
* `status`：长期状态，仅允许 `active / disabled / pending`
* `lock_until`：临时锁定截止时间
* `token_version`：强制使历史 token 失效
* `role_version`：角色变更后强制旧 token 失效

### 9.2 角色定义表 `roles`

```sql
CREATE TABLE roles (
    code VARCHAR(32) PRIMARY KEY,
    name VARCHAR(64) NOT NULL,
    category VARCHAR(32) NOT NULL,
    is_system BOOLEAN NOT NULL DEFAULT true,
    sort_order INTEGER NOT NULL DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

该表用于定义系统角色。

示例角色：

* T1
* T2
* T3
* ADMIN
* CUSTOMER

---

### 9.3 用户角色表 `user_roles`

```sql
CREATE TABLE user_roles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role_code VARCHAR(32) NOT NULL REFERENCES roles(code),
    is_primary BOOLEAN NOT NULL DEFAULT false,
    is_active BOOLEAN NOT NULL DEFAULT true,
    assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    assigned_by UUID NULL REFERENCES users(id),
    expires_at TIMESTAMPTZ NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (user_id, role_code)
);
```

字段说明：

* `is_primary`：用户主角色
* `is_active`：角色是否有效
* `expires_at`：临时授权角色的过期时间

### 9.4 当前会话角色表 `user_role_sessions`

```sql
CREATE TABLE user_role_sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID NOT NULL REFERENCES auth_sessions(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    active_role_code VARCHAR(32) NOT NULL REFERENCES roles(code),
    switched_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (session_id)
);
```

该表用于记录：

**当前登录会话正在使用的角色。**

设计原因：

* 一个用户可以拥有多个角色
* 不同浏览器会话可以选择不同角色
* 角色切换只影响当前会话
* 权限判断以该角色为准

````

### 9.3 会话表 `auth_sessions`

```sql
CREATE TABLE auth_sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    session_family_id UUID NOT NULL,
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    client_type VARCHAR(32) NOT NULL DEFAULT 'web',
    user_agent TEXT NULL,
    ip_address INET NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    revoked_at TIMESTAMPTZ NULL,
    revoke_reason VARCHAR(64) NULL
);
````

#### `status`

* `active`
* `revoked`
* `compromised`
* `expired`

### 9.4 Refresh Token 表 `refresh_tokens`

```sql
CREATE TABLE refresh_tokens (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID NOT NULL REFERENCES auth_sessions(id) ON DELETE CASCADE,
    token_family_id UUID NOT NULL,
    token_hash CHAR(64) NOT NULL UNIQUE,
    rotated_from_id UUID NULL REFERENCES refresh_tokens(id),
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    issued_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at TIMESTAMPTZ NOT NULL,
    used_at TIMESTAMPTZ NULL,
    revoked_at TIMESTAMPTZ NULL,
    revoke_reason VARCHAR(64) NULL
);
```

## 10. 明确功能边界

### 10.1 需要完成的功能

本模块必须完成：

1. 用户登录、登出
2. Access Token 签发与校验
3. Refresh Token 签发、轮换、撤销、reuse 检测
4. Cookie 安全设置
5. CSRF 防护
6. 用户角色识别
7. 角色级权限控制
8. 对象级权限校验能力接口
9. 登录失败限流
10. 账户临时锁定
11. 安全事件记录
12. 为审计模块提供操作者身份上下文

### 10.2 不需要完成的功能

本模块不负责：

1. 工单状态流转
2. 工单分配与升级
3. SLA 计时
4. 通知发送
5. Webhook 调用
6. KPI 统计
7. 知识库、报告、告警、SIM 数据查询
8. 配置中心具体业务配置内容
9. 前端页面权限展示逻辑的 UI 细节

### 10.3 暂时不做的功能

以下功能当前阶段不做：

1. 第三方 SSO（OIDC / LDAP / SAML）
2. MFA 多因素认证
3. 风险设备指纹识别
4. 异常地理位置登录检测
5. 验证码
6. 社交登录
7. 自助注册
8. 密码找回完整流程
9. 细粒度权限编辑器（仅先使用固定角色）
10. 多租户复杂组织树权限模型

## 11. 模块边界补充约束

1. 其他模块不得自行解析 token 并绕过统一鉴权中间件
2. 其他模块不得自己维护一套独立角色判断逻辑
3. 所有受保护接口必须统一接入本模块
4. 所有写接口必须在通过认证与授权后才允许执行
5. 所有业务模块必须从本模块获取当前用户身份，不允许信任前端传来的 `user_id`

## 12. 验收标准

### 12.1 认证验收

* 正确账号密码能登录成功
* 错误密码不能登录
* 用户不存在时不能登录
* 被禁用用户不能登录
* 被临时锁定用户不能登录

### 12.2 Token 验收

* Access Token 过期后不能继续访问
* Refresh Token 可正常轮换
* 旧 Refresh Token 再次使用会触发 reuse 检测
* 登出后 session 失效
* 改密后旧 token 全部失效

### 12.3 安全验收

* 密码哈希为 Argon2id
* 数据库不存明文密码
* 数据库不存明文 refresh token
* Cookie 属性符合设计要求
* 状态变更接口必须校验 CSRF
* 登录失败采用统一错误响应
* 三层限流和锁定规则生效

### 12.4 权限验收

* ADMIN 可访问管理接口
* CUSTOMER 不可访问配置接口
* 未认证请求访问受保护资源返回 401
* 已认证但无权限请求返回 403
* 对象级越权访问被拒绝

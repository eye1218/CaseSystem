# 工单核心模块

## 1. 模块功能说明与目标

### 1.1 模块定位

工单模块是系统的核心业务模块，负责管理工单对象从创建到关闭的完整生命周期。

本模块负责：

* 工单创建
* 工单查询与详情查看
* 工单更新
* 工单响应
* 工单处置
* 工单升级
* 工单关闭
* 工单重开（Reopen）
* 工单评论
* 工单分类引用
* 工单来源标记
* 工单过滤查询
* 工单操作日志
* 工单软删除

本模块不负责身份认证、权限体系实现、通知发送、Webhook 调用、Event 执行引擎、KPI 统计，但会为这些模块提供稳定的数据基础和触发点。

### 1.2 模块目标

本模块需要达成以下目标：

1. 建立统一且可扩展的工单数据模型
2. 用明确的主状态 / 子状态表达工单生命周期
3. 支持响应 SLA 与处置 SLA 对工单状态的影响
4. 支持升级给指定人和升级到池子两种模式
5. 支持手动建单和 API 建单
6. 支持组合过滤查询
7. 确保所有关键操作可审计、可追踪
8. 支持管理员软删除工单并提供回收站视图

## 3. 核心业务对象

### 3.1 Ticket

每一条工单代表一个需要处理的问题、事件或任务。

### 3.2 Ticket Category

工单分类，用于创建时归类、列表筛选和后续统计。

### 3.3 Ticket Pool

工单池，用于不同责任层级接收和领取工单。

### 3.4 Ticket Escalation

工单升级记录，用于记录升级申请、确认、拒绝及目标对象。

### 3.5 Ticket Comment

工单评论记录。

### 3.6 Ticket Action

工单操作日志，用于记录所有关键业务动作。

### 3.7 Ticket Create Request

工单创建幂等请求记录，用于防止重复创建。

| response_deadline_at   | 响应 SLA 截止时间 |
| resolution_deadline_at | 处置 SLA 截止时间 |
| responded_at           | 实际响应时间      |
| response_timeout_at    | 响应超时发生时间    |

| resolution_timeout_at  | 处置超时发生时间    |
| closed_at              | 关闭时间        |
| is_deleted             | 是否软删除       |
| deleted_at             | 软删除时间       |
| deleted_by             | 软删除执行人      |
| created_at             | 创建时间        |
| updated_at             | 更新时间        |

### 4.2 字段设计规则

1. 工单 ID 使用数字自增，不使用 UUID
2. `responded_at` 只要点击过“响应”就必须记录，不受是否超时影响
3. `resolved_at` 只要完成过处置就必须记录，不受是否超时影响
4. `response_timeout_at` 只在发生响应超时时记录
5. `resolution_timeout_at` 只在发生处置超时时记录
6. 工单在池中时，`assigned_to = NULL`
7. 工单被具体人领取后，`current_pool_code = NULL`
8. `source` 创建后不可随意修改

## 5. 主状态与子状态设计

## 5.1 主状态 `main_status`

主状态表示工单当前所处的主流程阶段。

### 1. `WAITING_RESPONSE`

待响应。
新工单创建后的初始状态。

### 2. `IN_PROGRESS`

处理中。
工单已完成响应，进入正常处理阶段。

### 3. `RESPONSE_TIMEOUT`

响应超时。
工单在响应 SLA 截止前未点击“响应”。

说明：

* 进入该状态后仍允许点击“响应”
* 点击“响应”后必须记录 `responded_at`
* 但主状态仍保持 `RESPONSE_TIMEOUT`
* 只要已记录 `responded_at`，就允许继续执行处置

### 4. `RESOLUTION_TIMEOUT`

处置超时。
工单已响应，但在处置 SLA 截止前未完成处置。

说明：

* 进入该状态后仍允许继续处置
* 处置完成后进入 `RESOLVED`

### 5. `RESOLVED`

已处置。
工单已经完成处置。

### 6. `CLOSED`

已关闭。
工单已结束。

### 7. `REOPENED`

已重开。
工单从关闭状态被重新打开的过渡状态。

说明：

* `REOPENED` 是短暂主状态
* 后续重新进入 `WAITING_RESPONSE`

---

## 5.2 子状态 `sub_status`

子状态表示工单当前附加的辅助流程状态，不替代主状态。

### 1. `NONE`

无子状态。默认值。

### 2. `ESCALATION_PENDING_CONFIRM`

升级待确认。
仅用于“升级给指定用户”时，等待目标用户确认。

### 3. `ESCALATION_CONFIRMED`

升级已确认。
目标用户已确认接收升级工单。

### 4. `ESCALATION_REJECTED`

升级被拒绝。
目标用户拒绝接收升级工单。

已确认规则：

* 主状态：`IN_PROGRESS`
* 子状态：`ESCALATION_REJECTED`

---

## 5.3 为什么采用主状态 + 子状态

采用双层状态模型的原因：

1. 工单主流程和升级流程是并行语义
2. 升级待确认、升级被拒绝更适合作为附加状态，而不是覆盖主流程
3. 响应超时和处置超时需要保留超时事实
4. 前端筛选、日志审计、Event 触发都需要更清晰的状态表达

---

## 6. 工单分类

### 6.1 当前分类模型

本期只支持一级分类。

### 6.2 默认分类

建议默认值：

* `SECURITY_INCIDENT`（安全事件）
* `SYSTEM_FAULT`（系统故障）
* `SERVICE_REQUEST`（服务请求）
* `ALERT_EVENT`（告警事件）
* `OTHER`（其他）

### 6.3 分类规则

1. 创建工单时必须指定分类
2. 分类可用于过滤查询
3. 后续支持管理员对分类进行增删改查
4. 停用分类不影响历史工单展示

### 7.1 来源枚举

```text
API
INTERNAL
CUSTOMER
```

### 7.2 含义

* `API`：通过 API 创建
* `INTERNAL`：内部人员手动创建
* `CUSTOMER`：客户手动创建

### 7.3 规则

1. 来源为必填
2. 客户创建工单时，来源必须标记为 `CUSTOMER`
3. 来源创建后不允许随意修改
4. 来源可作为查询和统计维度

## 9. 池子模型

## 9.1 固定池子

当前共有三个池子：

* `T1_POOL`
* `T2_POOL`
* `T3_POOL`

## 9.2 领取规则

1. T1 只能领取 `T1_POOL` 中的工单
2. T2 只能领取 `T2_POOL` 中的工单
3. T3 只能领取 `T3_POOL` 中的工单
4. ADMIN 可以领取所有池子中的工单

## 9.3 领取后的数据规则

* 领取前：

  * `assigned_to = NULL`
  * `current_pool_code = 对应池子`
* 领取后：

  * `assigned_to = 领取人`
  * `current_pool_code = NULL`

---

## 10. 升级模型

## 10.1 支持两种升级模式

### 模式一：升级给指定人

* 目标是明确的某个用户
* 需要该用户确认
* 会产生升级确认 / 升级拒绝结果

### 模式二：升级到池子

* 目标是某个池子
* 不需要特定的人确认
* 升级立即生效

---

## 10.2 T1 池默认升级规则

已确认规则：

* 如果工单当前位于 `T1_POOL`
* 用户点击升级时未指定具体人
* 系统自动升级到 `T2_POOL`

本期只固化这一条，其他层级默认升级规则暂不扩展。

---

## 10.3 升级重复点击处理

同一工单不允许并发产生多条未完成的“升级给指定人”的待确认请求。

规则：

1. 同一工单同一时间只允许存在一条 `ESCALATION_PENDING_CONFIRM` 升级请求
2. 重复点击升级不得重复创建待确认升级记录
3. 升级请求必须单独落表
4. 升级接口必须支持幂等控制

---

## 10.4 升级确认规则

### 升级给指定人

* 必须由指定用户确认
* 确认后进入升级已确认状态
* 拒绝后保留升级被拒绝子状态

### 升级到池子

* 不需要确认
* 直接进入目标池

---

## 11. SLA 规则（本模块内固化的消费口径）

### 11.1 SLA 归属

SLA 按客户配置。每个客户有自己的响应 SLA 和处置 SLA。

### 11.2 SLA 起算时间

统一从：

```text
ticket.created_at
```

开始计算。

### 11.3 响应 SLA

* 在截止时间前未点击“响应”，主状态变为 `RESPONSE_TIMEOUT`
* 记录 `response_timeout_at`

### 11.4 处置 SLA

* 在截止时间前未完成处置，主状态变为 `RESOLUTION_TIMEOUT`
* 记录 `resolution_timeout_at`

---

## 12. 业务流程

## 12.1 手动创建工单

### 支持角色

* CUSTOMER
* ADMIN
* T1
* T2
* T3

### 流程

1. 用户提交创建请求
2. 校验标题、描述、分类、优先级、风险分数、来源
3. 校验幂等键
4. 创建工单
5. 主状态设为 `WAITING_RESPONSE`
6. 子状态设为 `NONE`
7. 计算 `response_deadline_at`、`resolution_deadline_at`
8. 写入操作日志
9. 返回工单详情

---

## 12.2 API 创建工单

### 规则

1. API 创建也支持幂等键
2. API 允许重复事件
3. 即使业务内容重复，只要不是同一个幂等请求，也允许创建多个工单

### 流程

1. API 调用方提交请求
2. 校验接口身份
3. 校验分类、来源、风险分数等字段
4. 校验幂等键
5. 创建工单
6. 主状态设为 `WAITING_RESPONSE`
7. 写入操作日志

---

## 12.3 工单响应

### 可操作状态

* `WAITING_RESPONSE`
* `RESPONSE_TIMEOUT`

### 流程

1. 处理人点击“响应”
2. 校验权限
3. 记录 `responded_at`
4. 若当前主状态为 `WAITING_RESPONSE`，则变为 `IN_PROGRESS`
5. 若当前主状态为 `RESPONSE_TIMEOUT`，则主状态保持不变
6. 写入操作日志

### 结果

* 只要有 `responded_at`，即视为已执行响应动作

---

## 12.4 工单处置

### 可操作条件

允许处置的条件为：

* 主状态为 `IN_PROGRESS`
* 或主状态为 `RESOLUTION_TIMEOUT`
* 或主状态为 `RESPONSE_TIMEOUT` 且 `responded_at` 不为空

### 流程

1. 处理人点击“处置”
2. 校验权限和状态
3. 记录 `resolved_at`
4. 主状态变为 `RESOLVED`
5. 写入操作日志

---

## 12.5 工单关闭

### 可操作状态

* `RESOLVED`

### 流程

1. 客户或授权角色点击关闭
2. 校验权限
3. 记录 `closed_at`
4. 主状态变为 `CLOSED`
5. 写入操作日志

---

## 12.6 工单 Reopen

### 规则

* CUSTOMER 允许 Reopen
* Reopen 后重新进入待响应流程

### 流程

1. 客户点击 Reopen
2. 校验权限
3. 主状态置为 `REOPENED`
4. 清空 `assigned_to`
5. 写入操作日志
6. 后续重新流转到 `WAITING_RESPONSE`

---

## 12.7 升级给指定人

### 流程

1. 当前处理人发起升级
2. 校验升级权限
3. 校验当前不存在未完成待确认升级
4. 创建升级记录
5. 子状态置为 `ESCALATION_PENDING_CONFIRM`
6. 写入操作日志

### 确认

1. 指定用户点击确认
2. 更新升级记录状态为已确认
3. 子状态置为 `ESCALATION_CONFIRMED`
4. 更新责任层级和处理人
5. 写入操作日志

### 拒绝

1. 指定用户点击拒绝
2. 更新升级记录状态为已拒绝
3. 主状态保持 `IN_PROGRESS`
4. 子状态置为 `ESCALATION_REJECTED`
5. 写入操作日志

---

## 12.8 升级到池子

### 流程

1. 当前处理人发起升级
2. 校验升级权限
3. 目标为池子
4. 不需要确认
5. 清空 `assigned_to`
6. 设置 `current_pool_code = 目标池子`
7. 主状态保持 `IN_PROGRESS`
8. 子状态保持 `NONE`
9. 写入操作日志

---

## 12.9 池子领取

### 流程

1. 用户从池子中点击领取
2. 校验当前角色是否允许领取该池子
3. 将 `assigned_to` 设为当前用户
4. 清空 `current_pool_code`
5. 写入操作日志

---

## 12.10 软删除

### 规则

* 仅 ADMIN 可执行
* 仅允许软删除
* 软删除后普通用户不可见
* 管理员可在回收站视图查看

### 流程

1. ADMIN 发起删除
2. 校验权限
3. 设置 `is_deleted = true`
4. 记录 `deleted_at`、`deleted_by`
5. 写入操作日志

---

## 13. 幂等规则

## 13.1 手动创建幂等

手动创建工单必须支持 `idempotency_key`。

规则：

1. 同一用户 + 同一 `idempotency_key`，只能创建一次
2. 同一幂等键若对应不同请求体，必须报错
3. 内容去重只做提示，不做强拦截

## 13.2 API 创建幂等

API 创建也支持 `idempotency_key`。

规则：

1. 同一调用方 + 同一 `idempotency_key`，只能创建一次
2. API 允许重复事件
3. 只要不是同一幂等请求，就允许重复创建工单

---

## 14. 过滤查询设计

## 14.1 支持的过滤维度

1. 优先级
2. 分类
3. 风险分数
4. 创建时间
5. 主状态
6. 子状态

## 14.2 过滤规则

### 同一维度

使用 OR / IN。

例如：

* 优先级 = P1 或 P2
* 分类 = A 或 B

### 不同维度

使用 AND。

例如：

* 优先级 in (...)
* 分类 in (...)
* 风险分数 between ...
* 创建时间 between ...
* 主状态 in (...)
* 子状态 in (...)

这些条件组合生效。

## 14.3 风险分数过滤

支持区间筛选：

* `risk_score_min`
* `risk_score_max`

## 14.4 创建时间过滤

支持范围筛选：

* `created_from`
* `created_to`

## 14.5 状态过滤

### 主状态

支持按主状态多选过滤。

### 子状态

支持：

* `All`：不过滤子状态
* `No Sub Status`：筛选 `sub_status = NONE`
* 具体子状态值过滤

## 15. 数据库表设计

## 15.1 工单表 `tickets`

```sql
CREATE TABLE tickets (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    description TEXT NOT NULL,

    main_status VARCHAR(64) NOT NULL,
    sub_status VARCHAR(64) NOT NULL DEFAULT 'NONE',

    category_id BIGINT NOT NULL REFERENCES ticket_categories(id),
    source VARCHAR(32) NOT NULL,
    priority INTEGER NOT NULL DEFAULT 3,
    risk_score INTEGER NOT NULL DEFAULT 0,

    created_by UUID NOT NULL REFERENCES users(id),
    assigned_to UUID NULL REFERENCES users(id),
    current_pool_code VARCHAR(32) NULL REFERENCES ticket_pools(code),
    responsibility_level VARCHAR(16) NOT NULL,

    response_deadline_at TIMESTAMPTZ NULL,
    resolution_deadline_at TIMESTAMPTZ NULL,

    responded_at TIMESTAMPTZ NULL,
    response_timeout_at TIMESTAMPTZ NULL,

    resolved_at TIMESTAMPTZ NULL,
    resolution_timeout_at TIMESTAMPTZ NULL,

    closed_at TIMESTAMPTZ NULL,

    is_deleted BOOLEAN NOT NULL DEFAULT false,
    deleted_at TIMESTAMPTZ NULL,
    deleted_by UUID NULL REFERENCES users(id),

    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 15.2 分类表 `ticket_categories`

```sql
CREATE TABLE ticket_categories (
    id BIGSERIAL PRIMARY KEY,
    code VARCHAR(64) NOT NULL UNIQUE,
    name VARCHAR(128) NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT true,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 15.3 池子表 `ticket_pools`

```sql
CREATE TABLE ticket_pools (
    code VARCHAR(32) PRIMARY KEY,
    name VARCHAR(64) NOT NULL,
```

## 19. 验收标准

### 19.1 创建

* 工单 ID 使用数字自增
* 创建后主状态为 `WAITING_RESPONSE`
* 创建时必须带分类和来源
* 手动建单支持幂等

### 19.2 响应

* `WAITING_RESPONSE` 可响应并进入 `IN_PROGRESS`
* `RESPONSE_TIMEOUT` 可响应并记录 `responded_at`
* `RESPONSE_TIMEOUT` 响应后主状态仍保持不变

### 19.3 处置

* `IN_PROGRESS` 可处置
* `RESOLUTION_TIMEOUT` 可处置
* `RESPONSE_TIMEOUT` 且已记录 `responded_at` 时可处置
* 处置时必须记录 `resolved_at`

### 19.4 升级

* 升级给指定人时，需要确认
* 升级到池子时，不需要确认
* 重复点击升级不得产生重复待确认升级记录
* T1_POOL 未指定人升级时，自动进入 T2_POOL

### 19.5 池子

* T1 只能领取 T1_POOL
* T2 只能领取 T2_POOL
* T3 只能领取 T3_POOL
* ADMIN 可领取全部池子
* 领取后 `assigned_to` 生效，`current_pool_code` 清空

### 19.6 SLA

* SLA 从工单创建时间开始计算
* 响应超时后记录 `response_timeout_at`
* 处置超时后记录 `resolution_timeout_at`

### 19.7 过滤

* 优先级、分类、风险分数、创建时间、主状态、子状态均可过滤
* 同一维度多值使用 OR
* 不同维度组合使用 AND
* 子状态支持 `All` 和 `No Sub Status`

### 19.8 删除

* 仅 ADMIN 可删除
* 删除为软删除
* 普通用户不可见
* 管理员可在回收站查看
